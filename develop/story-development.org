* What a Story Is

A /story/ in this project is not documentation. It is a parallel
register — the same structural insights expressed through geography,
craft, and observation rather than through code and configuration.

Writing a story is a test of understanding: if a concept survives
translation into the valley's language, it is understood. If it
cannot — if the system vocabulary leaks through, if the geography
feels invented, if the tension dissolves — we do not yet understand
what we built. The translation is the test.

The project's stories are found documents from the transborder Western
Himalaya. Field notes, ledgers, logbooks, geological surveys, woven
records, testimony. Narrated by the Thread Walker, who observes and
records but does not explain what she does not understand.

* The Pipeline

#+begin_example
Research → Composition → Illustration → Publication
  │            │              │              │
  ▼            ▼              ▼              ▼
surveys/     draft.org      generate_     website/content/
research.org  → .md        images.py     writing/<slug>.md
                                          website/static/
                                          images/writing/<slug>/
#+end_example

Each phase produces artifacts. The workspace holds development
artifacts; the Hugo tree holds publication artifacts. The slug
is identical across all locations.

** Phase 1: Research

Domain-wide surveys in =domains/epistem/surveys/=. Per-story research
in =stories/<slug>/research.org=. The survey serves many stories; the
=research.org= is one story's lens on the survey.

When a story requires a survey, the survey is a separate WP
(e.g., WP-0012 for the Diamer survey). The story WP depends on the
survey WP in the dispatch plan.

** Phase 2: Composition

The sutradhar uses =summarize-corpus= (or direct source material) and
=story-compose= to draft the story. The draft may live as =draft.org=
in the workspace, or go directly to Hugo markdown.

The human reviews, annotates from lived experience, and edits. Voice
calibration: read at least two existing stories before drafting. The
most recent is the strongest reference.

** Phase 3: Illustration

A single =generate_images.py= in the workspace. PEP 723 inline script
metadata, run with =uv run generate_images.py=. Output directed to
=website/static/images/writing/<slug>/=.

** Phase 4: Publication

Story as Hugo markdown at =website/content/writing/<slug>.md=. Set
=draft = true= until human approval. Deploy with =./deployment/deploy.sh hugo=.

* The Story Workspace

#+begin_example
stories/                              # story development workspaces
  <slug>/                             # one per story
    notes.org                         # origin, plan, context, voice refs
    keys.org                          # translation keys this story introduces
    generate_images.py                # PEP 723 illustration script
    research.org                      # (optional) per-story research
    draft.org                         # (optional) working draft
#+end_example

The =<slug>= matches the Hugo slug exactly — the same string used in
=website/content/writing/<slug>.md= and =website/static/images/writing/<slug>/=.
No mapping, no renaming.

** Required files

*** =notes.org=

The development record. Based on the pattern that emerged in
=instrument-makers-rest/session-notes.org= (story #4) — the only story
with a proper handoff document. Sections:

- *Origin* — what prompted this story, which sessions or surveys feed it
- *Story Plan* — section outline, concept mapping, document type
- *Technical Context* — what the writing agent needs to understand
- *Voice References* — which existing stories to read for calibration
- *Continuity Notes* — characters, places, callbacks to other stories
- *Illustration Notes* — visual register, palette, image list
- *WP Reference* — link to =workpacks/NNNN-slug.org= (if story has a WP)

Not all sections need to be long. A story that emerges in a single
session may have a one-paragraph origin and no technical context. But
the skeleton is stable.

*** =keys.org=

Translation keys introduced or refined by this story. Org table:

#+begin_example
| System Concept | Story Equivalent | Notes |
|----------------+------------------+-------|
| ...            | ...              | ...   |
#+end_example

These feed the global translation table (below). The per-story file
is the provenance record; the global table is the union.

*** =generate_images.py=

The illustration script. Convention:

#+begin_src python :tangle no
from pathlib import Path

SLUG = Path(__file__).parent.name
OUT = (Path(__file__).parent / "../../website/static/images/writing" / SLUG).resolve()
OUT.mkdir(parents=True, exist_ok=True)
#+end_src

Two levels up from =stories/<slug>/= reaches the project root. The slug
is derived from the directory name — no hardcoding.

** Optional files

*** =research.org=

Per-story research notes. For stories with a domain survey, this file
links to the survey and adds story-specific annotations:

#+begin_example
See [[file:../../domains/epistem/surveys/diamer-nanga-parbat.org][Regional Survey: Diamer and the Nanga Parbat Massif]] (WP-0012).
#+end_example

When the story requires original research beyond the survey, the
notes go here.

*** =draft.org=

Working draft in Org before conversion to Hugo markdown. Early stories
(1–6) kept drafts in the workspace; later stories composed directly
into Hugo format. Either approach is valid.

* Story Work Packages

A story WP follows the standard convention (=develop/work-packages.org=)
with these additions in the Specification section:

- Geographic setting (valley, altitude, season)
- Document type (field notes, logbook, survey, testimony)
- Technical substrate (what system concept is being translated)
- Translation keys to establish
- Illustration count and visual register
- Voice references (which existing stories to read)
- Continuity constraints (characters, callbacks, geography)

Trust level is typically *pair* (human and agent co-compose) or
*checkpoint* (agent drafts, human reviews before publish).

Story WPs live in =workpacks/= with the standard =NNNN-slug.org= naming.
They are not special — they follow the same lifecycle. The workspace's
=notes.org= links to the WP by number.

Stories composed in-session (without a WP) simply omit the WP link —
the convention accommodates both modes.

* Commit Convention

"Chunk commits by deployability." A story commit includes:

- =stories/<slug>/notes.org=
- =stories/<slug>/keys.org=
- =stories/<slug>/generate_images.py=
- =website/content/writing/<slug>.md=
- =website/static/images/writing/<slug>/*.png=

All in one commit. The workspace is the development record; the Hugo
files are the deployable artifact.
