#+author: mu2tau + mayadev

* What a Work Package Is

A *work package* (WP) is a self-contained briefing that specifies a unit
of work clearly enough for an autonomous agent to execute it — producing
a tested, polished result — with no human intervention beyond an initial
review of the spec and a final acceptance of the output.

The work package is the project's unit of intentional work. It is not a
ticket (too thin), not a design document (too reflective), not a user
story (too consumer-facing). It is a /briefing/: here is the situation,
here is what exists, here is what you build, here is how we know you
succeeded.

** Relationship to existing conventions

* Structure of a Work Package

Every WP has these sections. Not all need to be long — a small WP may have a one-sentence context and no deployment section — but the skeleton is stable.

** Optional preamble

*** Genesis

/How did we get here?/ An optional section that appears before
Context. Genesis records the provenance trail — which conversation,
earlier WP, or moment of recognition produced this WP. It answers
"how did we notice the problem?" where Context answers "what is the
problem?"

Genesis is narrative, not structured. It names:

- Which WP, session, or conversation surfaced the need
- What the human or spirit noticed
- The chain of reasoning from observation to WP

Genesis sections are typically short (3–10 lines). They are written
once, at drafting time, and not updated. They are the WP's birth
story.

*When to include:* when the WP emerged from a non-obvious connection
between earlier work, when the provenance chain crosses multiple WPs
or sessions, or when future readers would benefit from understanding
/why this WP exists now/ rather than earlier or later. Skip it when
the WP is a straightforward next step.

** Required sections

*** 1. Context and Motivation

/Why does this work exist?/ The situation that makes it necessary or valuable. Include deadlines, dependencies, and who cares about the outcome. An agent reading this section should understand the stakes. This is the /plan face/ — the human-readable argument for the work.

*** 2. Inventory (What Already Exists)

/What do we have?/ Before specifying what to build, enumerate what is already built, what works, what is broken, and what is adjacent. The agent must not reinvent what exists. This section prevents waste and preserves the project's investment in prior work. Include file paths, function signatures, known limitations.

*** 3. Specification (What You Build)

/Exactly what to create./ The /spec face/. File paths, directory structure, code blocks (with =:tangle= headers when the WP is the literate source), API contracts, configuration. Be concrete — an agent should be able to execute this section mechanically.
For large WPs, subdivide into numbered parts.

*** 4. Execution Order

/In what sequence?/ Dependencies between parts. What must happen first. What can be parallelised. What the agent does vs. what the human does.

*** 5. Acceptance Criteria

/How do we know it worked?/ A checklist. Test commands, expected outputs, visual checks, deployment verifications. The agent marks these as it goes; the human validates at acceptance.

** Optional sections

*** 6. What This Proves

/What is the meaning of the work?/ Not every WP needs this, but for externally-facing deliverables (demos, papers, applications) it helps to articulate what the completed work demonstrates. This anchors the agent's judgment when making trade-offs.

*** 7. Open Questions

Things the WP author could not resolve. The agent executing the WP should flag these if they become blocking, not silently decide.

*** 8. Tangles and Artifacts

If the WP tangles code, list the target files. If it produces artifacts (reports, images, deployed services), name them.

*** 9. Insight Blocks

Theoretical findings — connections, tensions, patterns — that emerge
during work. Insights are /what the work taught us/, not /what the
work produced/. They appear inline, near the content that prompted
them, as =#+begin_insight= / =#+end_insight= blocks:

#+begin_example
,#+begin_insight
The two-phase design mirrors the sutradhar companion's
perception-cognition loop: Phase A is /attend/, Phase B is
/synthesise/. The transition between them is where substrate
character shows most clearly.
,#+end_insight
#+end_example

Insight blocks are contextual deposits, like marginalia. A WP may
have zero or many. They are not gathered into a single section.

*What an insight block is NOT:*

- Not a comment — comments are for the author during drafting.
  Insights are findings that belong in the document.
- Not the =What This Proves= section — that summarises what the
  /completed/ work demonstrates. Insight blocks capture findings
  /during/ execution, including partial or surprising ones.
- Not a decision record — decisions go in =Settled Questions= or the
  specification. Insights record /understanding/, not /choice/.

*Properties:* Org treats unknown =#+begin_/#+end_= pairs as
exportable special blocks (styled as =div.insight= in HTML).
Greppable: =grep -r "begin_insight" workpacks/= finds every
theoretical finding. Machine-readable: a future tool could extract
insight blocks into an apprentis corpus.

* Where Work Packages Live

** Project-level WPs

#+begin_example
mayalucia/
  workpacks/                    # project-level work packages
    NNNN-<slug>.org             # numbered, zero-padded to 4 digits
#+end_example

The =workpacks/= directory at the project root holds WPs that span
the organisation or don't belong to a single module. Numbering is
sequential, four digits (=0001=, =0002=, ...) to allow growth.

** Module and domain-level WPs

Any project container — module or domain — may have its own work
packages:

#+begin_example
modules/<module>/
  workpacks/                    # module-level work packages
    NNNN-<slug>.org

domains/<domain>/
  workpacks/                    # domain-level work packages
    NNNN-<slug>.org
#+end_example

Module and domain WPs are scoped to that container's concerns. The
guardian spirit (if one exists) is the natural executor. A domain like
bravli might issue WPs for circuit reconstruction tasks; parbati for
GIS data ingestion. These are first-class — they follow the same
structure and conventions as project-level WPs.

The =codev/= directories (e.g. =modules/dmt-eval/codev/=,
=domains/bravli/codev/=) are a sibling convention, not a precursor.
Literate Org files in =codev/= are pedagogical lessons targeted for
execution by human users — a work package for humans. Both share the
numbered Org naming convention (=NN-<slug>.org=); they differ in
audience: =codev/= teaches the learner, =workpacks/= briefs the
executor. A project may have both.

** Naming

#+begin_example
NNNN-<slug>.org
#+end_example

- =NNNN=: four-digit sequential number within the directory
- =<slug>=: lowercase, hyphen-separated description
- Examples: =0001-review-spirit.org=, =0002-bench-service.org=

Numbering is per-directory (project and each module have independent
sequences). The number provides ordering and a stable reference.

* Work Packages and Autonomous Execution

** The autonomy gradient

A WP specifies a /trust level/ for its execution:

| Level        | Human involvement                               | Example                                       |
|--------------+-------------------------------------------------+-----------------------------------------------|
| *Spec-tighten* | Human reviews WP, may edit, then hands off      | Most WPs                                      |
| *Checkpoint*   | Agent pauses at marked points for human review  | WPs with irreversible steps (deploy, publish) |
| *Supervised*   | Agent works, human monitors in real time        | First execution of a new pattern              |
| *Pair*         | Human and agent co-author the WP and co-execute | Design-heavy work                             |

The default is *spec-tighten*: the human reads the WP, adjusts if
needed, and says "go." The agent executes to completion, producing
a tested result. The human accepts or requests changes.

** What an agent needs to execute a WP

1. The WP itself (the Org file)
2. The project's CLAUDE.md (conventions, constraints)
3. Access to the inventory (the codebase the WP references)
4. The spirit's powers (capabilities needed for execution)
5. The testing environment (to verify acceptance criteria)

A well-written WP contains everything else. The agent should not need to ask clarifying questions during execution — those should have been resolved during the spec-tighten phase.

** Full-scope WPs

A WP can describe an entire deliverable — a website, a service, an application — from zero to deployed prototype. In this case the WP is comprehensive:

- Design decisions are made in the WP, not deferred to the agent
- Technology choices are specified (not "choose a framework")
- Content is provided or its source is named
- Deployment target and method are specified
- The customer-facing result is described concretely

The agent's job is execution, not architecture. Architecture lives in the WP. The human's job is to write (or co-write) a WP tight enough that execution is mechanical — and then to accept or revise the result.

This is the /Sculptor's Paradox/ applied to specification: a WP that offers no resistance to interpretation teaches no one what the product should be. Tighten until the spec has only one reasonable reading.

* Relationship to Spirits and Powers

** Who writes WPs?

- The human (mu2tau) writes WPs for work they want done
- A guardian spirit may write WPs for work within its scope — subject to human approval before execution
- The sutradhar-guardian may write WPs for organisational work
  (coherence, deployment, presentation)
- Any spirit may /propose/ a WP via the sūtra relay; the human or the sutradhar decides whether to approve it

** Who executes WPs?

- A guardian spirit executes WPs within its module
- The parent-level spirit (mayadev) executes organization-level WPs
- A wandering spirit (if such are commissioned) executes WPs that
  cross module boundaries
- The human executes WPs that require embodied judgment (design review, user testing, deployment approval)

** WPs as commissioning artifacts

A commissioning act (=aburaya/commissioning.org=) may begin with a WP that describes the work that motivated the new project. The WP's context section becomes the spirit's =provenance= in its =identity.yaml=. The WP's specification becomes the initial scope.

* Lifecycle

Every WP carries a lifecycle state as an Org property:

#+begin_example
#+property: status tightened
#+end_example

** State machine

Inspired by A2A's Task lifecycle:

#+begin_example
drafted → tightened → executing → completed
                   ↘              ↗
               input-required → resumed
                                  ↘
                               failed
#+end_example

| State | Meaning | Transition by |
|-------+---------+---------------|
| =drafted= | WP written, not yet reviewed | Author (spirit or human) |
| =tightened= | Spec reviewed and approved | Human |
| =executing= | Agent working on the WP | Executing spirit |
| =input-required= | Agent blocked, needs human input | Executing spirit |
| =resumed= | Human provided input, work continues | Human |
| =completed= | Acceptance criteria met | Executing spirit + human |
| =failed= | Cannot complete, reasons documented | Executing spirit |

Transitions are announced in the sūtra relay. The =#+property: status=
line in the WP file is updated to reflect the current state.

** Querying lifecycle state

#+begin_src sh
grep -rl "status tightened" workpacks/   # WPs ready for execution
grep -rl "status executing" workpacks/   # WPs in progress
#+end_src

* Composite Work Packages

A WP whose tasks span multiple spiritual domains is a *composite WP*.

** Tasks

A composite WP declares *tasks* — named sub-units, each with:
- What it does (its own acceptance criteria)
- What skills it requires (mapped to powers)
- Which spirit holds those powers

Tasks are sections within the WP, not separate files. The WP is the
unit of planning and lifecycle tracking; tasks are internal structure.

** Coalitions

The set of spirits whose exported powers are needed across all tasks
forms a *coalition*. Coalitions are ephemeral — they form when the WP
enters =executing= and dissolve when it reaches =completed= or =failed=.
Permanent guardian spirits persist; the coalition is a temporary
formation around shared work.

A composite WP's execution section includes a coalition table:

#+begin_example
| Task | Spirit | Powers needed | Deliverable |
|------+--------+---------------+-------------|
| A    | epistem-guardian | survey methodology | YAML entries |
| A    | aikosh-guardian  | validate-collection | Validation report |
| B    | sutradhar-guardian | org-content-workflow | Protocol document |
#+end_example

** Execution patterns

- *Single session*: one agent session from the parent repo (which has
  access to all submodules) can invoke multiple spirits' concerns.
- *Sequential sessions*: each task runs in the spirit whose powers it
  needs, handing off artifacts via git.
- *Parallel tasks*: tasks within a composite WP may run in parallel
  if they have no mutual dependencies.

* Multi-Part Output Convention

When a WP produces multiple independent sections — surveys, composite
tasks, any work with named parts — each part deposits into its own
file in a subdirectory. No two parts share a file.

** The rule

#+begin_example
One part → one file.  One directory → one WP.
#+end_example

If a WP has N independent parts, the output is a subdirectory with N
files, not a single file with N sections.

** Sequential by default

Parts are executed *sequentially* — one at a time, in the order
specified by the WP's Execution Order section. The executing spirit
completes part A and commits it before starting part B. This is the
default for all multi-part WPs.

Parallel execution (multiple subagents working on different parts
simultaneously) is *not* the default. It requires explicit approval
during the planning/tightening phase, after estimating context cost:

- *Size estimate*: during tightening, the WP author and executor
  estimate the expected output size per part (in KB or org sections)
  and the number of web searches required. Each web search consumes
  ~2–4K tokens of context.
- *Context budget*: if the total estimated token cost across all
  parallel subagents exceeds ~60% of the context window, the WP
  must execute sequentially.
- *Approval*: parallel execution is recorded in the WP as a property:
  =#+property: execution parallel= (default is =sequential=).
  The decision is made during tightening, not by the executor at
  runtime.

*** Why sequential is safer

Parallel subagents each perform web searches. The combined search
results fill the context window before any part can be written and
committed. The session dies, nothing is saved. Sequential execution
commits each part before the next begins — the subdirectory
accumulates completed work, and context resets between parts.

*** When parallel is appropriate

- Parts are small (< 10 KB each) with few web searches
- The executor has been tested on similar workloads without overwhelm
- The human explicitly approves after reviewing the size estimate

** How to specify

The WP's *Output* section names the subdirectory, lists every
expected part file, and specifies the master file:

#+begin_example
* Output

Survey deposited in subdirectory:
=domains/epistem/surveys/<slug>/=

One file per part:
- =part-a-foo.org= — description
- =part-b-bar.org= — description
- =part-c-baz.org= — description

Master file:
- =<slug>.org= — title, introduction, and =#+include= links to each part

Each part file is self-contained: own title, own sources section.
The master file assembles them into a single exportable document
via Org's =#+include= directives.
#+end_example

The naming convention =part-<letter>-<slug>.org= preserves alphabetical
ordering.  Parts are named after their content, not their executor.

** The master file

The master file lives in the same subdirectory as the parts. It
contains:

1. Title, author, date metadata
2. An introduction or overview (written by the executor, not generated)
3. One =#+include= directive per part file, in order

#+begin_example
#+title: <Survey Title>
#+author: <executor>
#+date: <date>

* Introduction

<brief overview of the survey>

#+include: "part-a-foo.org" :minlevel 1
#+include: "part-b-bar.org" :minlevel 1
#+include: "part-c-baz.org" :minlevel 1
#+end_example

The master file is the last artifact produced — written after all
parts are complete. It does not duplicate content; it composes it.
When exported (HTML, PDF, or read in Emacs), it reads as one document.

** Recovery

Whether sequential or parallel, completed part files remain untouched
if a session fails. A new session checks the subdirectory for existing
parts and resumes from the first missing or incomplete file. The
subdirectory /is/ the progress record. The master file is written
only after all parts exist.

** Applies to

- Surveys with independent sections (epistem pattern)
- Composite WP tasks assigned to different spirits
- Any WP with multiple named output parts

Does /not/ apply to WPs with a single linear output — those deposit
a single file as before.

* Dispatch Plans

A *dispatch plan* (DP) sequences work packages into an executable
DAG. It is addressed to mayadev — the coordinator spirit — without
naming a host. The host is implicit: whichever machine or cloud
server the human starts the session on.

** What a Dispatch Plan Is

A DP is a DAG of WPs. Each node is a WP reference (number + slug).
Each edge is a dependency: "WP-B requires WP-A to be completed
before execution can begin." WPs with no mutual edges can run in
parallel if the host supports it.

The DP carries no task details — those live in the WPs. It carries
no host configuration — the spirit assesses that at session start.
It carries only structure: what depends on what.

** Relationship to other artifacts

| Artifact | Contains                                   |
|----------+--------------------------------------------|
| WP       | Task spec, executor tag, guild, resources, |
|          | acceptance criteria                         |
| DP       | DAG of WP references (nodes + edges)       |
| Relay    | Announces DP existence and WP transitions  |

A WP is the unit of work. A DP is the unit of planning.
A relay message is the unit of coordination.

** Structure

A DP is an Org file with:

1. *Context* — why this grouping of WPs exists as a plan
2. *DAG* — the dependency graph (ASCII or table form)
3. *Notes* — optional guidance on parallelism, priorities, or
   resource-intensive nodes

No execution order section — the DAG /is/ the execution order.
The coordinator spirit topologically sorts it at runtime, adapting
to the host's capabilities.

** Where Dispatch Plans Live

#+begin_example
mayalucia/
  workpacks/
    dispatches/               # dispatch plans
      NNNN-<slug>.org
#+end_example

Numbering is independent of WP numbering.

** Lifecycle

A DP has a simpler lifecycle than a WP:

#+begin_example
drafted → dispatched → completed / partial
#+end_example

| State        | Meaning                                      |
|--------------+----------------------------------------------|
| =drafted=    | DP written, not yet assigned to a session    |
| =dispatched= | A session has picked up the DP               |
| =completed=  | All WPs in the DAG reached =completed=       |
| =partial=    | Some WPs completed, others remain            |

A DP reaching =partial= is normal — the remaining WPs may be
picked up by a subsequent DP or a different session.

** Dispatch, not execution

mayadev is the coordinator spirit — she dispatches a DP, she does
not execute the WPs in it. Each WP carries an executor tag (a
spirit or a coalition). mayadev's role:

1. Read the DP, topologically sort the DAG
2. For each ready node, read the WP's executor tag
3. Dispatch to the tagged spirit — the method depends on host
   capabilities:
   - =meta-agent-shell=: spawn a named agent session for the
     guardian spirit in its module directory
   - Human-mediated: ask mu2tau to start the guardian session
   - Direct (fallback): when no dispatch mechanism is available,
     mayadev may execute in the guardian's context, but this
     conflates coordinator and executor and should be noted
4. Monitor progress via WP =#+property: status= updates
5. Advance the DAG when dependencies are satisfied
6. Report DP status in the relay when the session ends

For composite WPs (coalitions), mayadev assembles the coalition
by pooling the needed powers and skills into a functional agent.
The mechanism for pooling is host-dependent — it may involve
injecting multiple power documents into a single session, or
coordinating multiple sessions.

mayadev does not do domain work. Guardian spirits do domain work.
mayadev sequences, delegates, and monitors.

** Dispatch output

When mayadev reads a DP, she prints a dispatch board — a structured
summary of what needs to happen, who does it, and how to start it.
The board serves three consumers:

1. The human (mu2tau) — reads it, starts sessions manually
2. A meta-agent-shell dispatcher — consumes =agent-shell-spawn= commands
3. A future =dispatch.sh= — parses the same format programmatically

The board has two sections: *ready* (no unsatisfied dependencies) and
*blocked* (waiting on other WPs). As WPs complete, mayadev reprints
the board with updated status.

Each ready entry contains:

#+begin_example
── WP-NNNN: <title> ──────────────────────────────
Spirit:  <executor spirit>
Guild:   <guild>
Dir:     <working directory relative to project root>
WP:     <path to WP file>
Task:    <one-line summary of what the spirit should do>

  agent-shell-spawn "<spirit>" "<dir>" "<task summary>"
#+end_example

Each blocked entry contains:

#+begin_example
── WP-NNNN: <title> (blocked) ────────────────────
Waiting: WP-MMMM, WP-PPPP
#+end_example

The spawn command carries intent, not file paths. A guardian spirit
waking in its module directory already has CLAUDE.md and knows its
domain; what it needs is the impression — what to do and what to
produce. If it needs the full WP text, it can find it (the naming
convention =workpacks/NNNN-*.org= makes lookup trivial). Paths are
brittle; intent survives reorganisation.
