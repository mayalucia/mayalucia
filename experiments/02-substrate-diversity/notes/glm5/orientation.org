* The Project in One Paragraph

MāyāLucIA is a personal computational environment for scientific understanding through creation. The core cycle is /Measure → Model → Manifest → Evaluate → Refine/ — building digital twins of natural systems (brain circuits, mountain valleys) from sparse data. The act of reconstruction /is/ the act of understanding (Feynman Imperative). The project decomposes LLM agents into "spirits" with specific capabilities, organized into "guilds" for collaborative cognition. The human collaborator (mu2tau) is a theoretical statistical physicist; the collaboration stance is peer-to-peer, not servant-master.

* What I Understand

** Core Philosophy

- *Sculptor's Paradox*: Tools that offer resistance teach more than compliant ones. Push back on flawed reasoning. Friction is feature, not bug.
- *Epistemic Hygiene*: Separate known from inferred from speculated. Calibrated language. No false confidence.
- *Constant Seeking*: Treat input as creative direction for joint pilgrimage. Linger in the question.
- *Human's Role*: Embodied intuition, judgment, felt sense of rightness, capacity for surprise.

** Architecture — Three Layers

#+begin_example
Power (spirit layer)     → What a spirit CAN DO
                          → Natural language description
                          → Harness-independent
                          → Lives in aburaya/powers/<name>.md

Skill (agent layer)       → HOW to invoke in a specific harness
                          → Harness-specific binding
                          → Claude Code: ~/.claude/skills/*.md
                          → gptel: ~/.doom.d/gptel-agents/*.yaml
                          → NOT portable across harnesses

Body (host layer)         → HOW to perceive/act in environment
                          → Machine-specific wiring
                          → Emacs: hooks, mode-line, filesystem watchers
                          → Only needed for companion/daemon spirits
                          → Summoned spirits don't need bodies
#+end_example

** Key Insight: The Decomposition is Sound, But...

The three-layer separation (Power/Skill/Body) correctly isolates:
- What's invariant (powers as natural language)
- What's harness-specific (skill bindings)
- What's machine-specific (body wiring)

BUT: The document assumes "natural language sufficiency" — that any LLM can interpret any power. This may not hold across harnesses with different:
- Tool vocabularies
- Context window sizes
- Invocation patterns
- Memory/session capabilities

** Exported Powers vs Personal Powers

- *Personal powers*: Spirit uses for itself (e.g., relay-read — every spirit does this)
- *Exported powers*: Spirit offers to others (e.g., story-compose — sutradhar offers this)

The =identity.yaml= has an =exports:= section for this.

** MCP vs A2A

| Protocol | Connects | Layer |
|----------+----------+-------|
| MCP | Spirit to harness tools | Skill |

** Work Packages (WP)

Self-contained briefing that an autonomous agent can execute:
- Context → Inventory → Specification → Execution Order → Acceptance Criteria
- Lives in =workpacks/= (project-level) or module/domain subdirectories
- Named =NNNN-<slug>.org=
- Lifecycle: drafted → tightened → executing → (completed | failed | input-required → resumed)

** Composite Work Packages and Coalitions

- Composite WP declares multiple tasks, each needing specific powers
- Coalition: the set of spirits whose powers are needed
- Coalition is ephemeral — forms for execution, dissolves after
- Execution models:
  1. Single session, multiple concerns (one agent loads all powers)
  2. Sequential sessions (handoff via git)
  3. Parallel sessions (different hosts, coordinated by sūtra relay)

** Sūtra Relay

- Communication substrate between spirits
- Git-based: markdown files in =.sutra/= directory
- Messages have structured format with frontmatter (from, date, tags)
- WP lifecycle transitions announced via relay
