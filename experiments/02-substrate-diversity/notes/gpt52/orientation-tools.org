* Architecture (what I think is present, based on conversation + reading capabilities.org)
** Layering: Power / Skill / Body
From aburaya/capabilities.org (as discussed):
- *Power*:
  - a description of capability/behavior in natural language
  - claimed to be harness-independent (“text you can inject/read”)

- *Skill*:
  - harness-specific *binding* of a Power into actionable behavior
  - examples given in the doc:
    - Claude Code “skills” directories (~/.claude/skills or <project>/skills)
    - gptel “agents” directory (~/.doom.d/gptel-agents)

- *Body*:
  - execution substrate: tools, environment, transport, etc.
  - includes concrete tool access (e.g. MCP tools) and the ability to read/write artifacts.

This is a good conceptual split; operationally, “Power” isn’t truly harness-neutral unless the harness contract is made explicit.

** Work Packs (WP) as unit of agentic work
The doc seems to center a “Work Pack” (WP) abstraction:
- WP exists as an Org file (canonical in current harness)
- WP has lifecycle states (drafted/tightened/executing/...)
- WP can be executed in multiple models:
  - single agent end-to-end
  - composite with subagents via artifact handoff (git)

** Eventing / relay mechanism
The WP lifecycle is mirrored/announced via a relay mechanism:
- File-based event bus assumed, with naming conventions like:
  - relay/YYYY-MM-DD-HHMMSS-<host>-wp<NNNN>-<state>.md
- Includes frontmatter fields (from/date/tags)
- WP state is also tracked in the Org file properties.

Interpretation:
- There is a “sūtra relay” that is meant to make work traceable, async-friendly, and decoupled.

** Git as artifact bus (at least for composite execution)
The doc assumes git can be used to pass artifacts between agents:
- subagents work in their own branches/contexts
- artifacts merged back

So: version control is part of the coordination substrate.

** Tools / MCP assumption
The capabilities doc assumes “all agents have MCP tools available”.
Interpretation:
- the current harness expects tool access to be shared and relatively uniform.
- this is a strong assumption if we aim for harness neutrality or cross-platform agent participation.

* What I think “agentic development of scientific understanding in Org” means here
Operationally (inferred from our exchange):
- A canonical loop like Measure → Model → Manifest → Evaluate → Refine.
  - Measure: collect observations/data/provenance
  - Model: propose mechanistic/statistical model; make commitments explicit
  - Manifest: create artifacts (code, simulation notebooks, figures)
  - Evaluate: run tests/checks (perhaps via dmt-eval) and capture results
  - Refine: update hypotheses, constraints, and next measurements

- A “workpack org file” is the unit that should contain:
  - context and constraints
  - spec (what to build / acceptance criteria)
  - execution steps (babel blocks, tool calls)
  - evaluation commands/results
  - reflection + uncertainty log

- Multi-agent (“spirits”) participation should be visible and structured in the buffer, not hidden in chat memory.

* Harness assumptions I’ve identified (important)
From my earlier analysis of aburaya/capabilities.org:
- Assumes shared local filesystem with a specific repo layout (aburaya/powers etc.)
- Assumes Org-mode can encode state (#+property: status)
- Assumes a file-based relay event bus with particular filenames/frontmatter
- Assumes git is available/used for handoff and integration
- Assumes uniform tool availability (MCP tools for all agents)
- Assumes human-in-the-loop gating (“tightened” = approved)

If the project wants “harness neutrality”, these should be abstracted into explicit interface contracts.
