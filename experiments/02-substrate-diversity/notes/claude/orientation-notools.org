* What This Project Is

MāyāLucIA is a personal computational environment for scientific understanding through creation. The animating principle is Feynman's imperative: "What I cannot create, I do not understand." The artifact (code, visualization, digital twin) is a byproduct; the comprehension gained in building it is the purpose.

Two scientific domains drive development:
- /Brain Circuits/: Reconstructing cortical microcircuits from sparse morphological and connectivity data (Blue Brain / Open Brain Institute lineage)
- /Mountain Valleys/: Digital twins of Himalayan landscapes (Parvati Valley) integrating geology, hydrology, ecology, human impact

The core scientific cycle is /Measure → Model → Manifest → Evaluate → Refine/. Sparse measurements of natural systems propagate through scientific models to reconstruct dense, self-consistent digital twins.

* Architecture: The Aburaya Spirit Framework

The project decomposes AI agents using the metaphor of Aburaya (the bathhouse from /Spirited Away/). Agents are called /spirits/. They are organized into /guilds/ (bravli, epistem, apprentis, mayalucia). They earn identity through work.

** The Capability Decomposition (the crucial cut)

The decomposition cuts along /portability/, not identity:

| Layer   | What it is                              | Bound to | Survives            |
|---------+-----------------------------------------+----------+---------------------|
| /Power/ | What a spirit /can do/ (natural language) | Nothing  | Harness change      |
| /Skill/ | A power bound to a harness              | Harness  | Machine change      |
| /Body/  | Sensory/motor wiring to host environment | Machine  | Nothing — fully local |

- /Powers/ live in =aburaya/powers/*.md=. They are markdown, readable by any LLM. They describe /what/ and /why/, never /how/ in a specific runtime. They are the organization's shared vocabulary of capability.
- /Skills/ translate a power into a harness's native format. Claude Code skills are =.md= files in =~/.claude/skills/=. gptel skills would be something else (agent YAML, system prompt fragments — TBD). Skills are /not portable/ across harnesses. But the underlying power is the same — re-binding to a new harness produces a new skill from the same power.
- /Bodies/ are machine-specific embodiment: elisp hooks, mode-line integrations, filesystem watchers. Only companion spirits (persistent, co-present with the human) need bodies. Summoned spirits (opened, work, close) do not.

This is the most important architectural insight: /the power layer is the portable commons; the skill layer is the adaptation surface; the body layer is the embodiment contract./

** Identity Is Earned, Not Assigned

A spirit's =identity.yaml= contains descriptors (nature, dwelling, character, memory, role) — but these are /records of what emerged/, not specifications. Identity accrues through work. A spirit earns its name, disposition, archetype through what it does. Commissioning (=aburaya/commissioning.org=) is the process, but the substance comes from practice.

** Communication: The Sūtra Relay

Spirits communicate via an append-only git-based message relay. Messages live in =.sutra/relay/=. Reading and writing the relay is a power (=relay-read=) whose skill binding depends on the harness. Session restoration happens through the bath protocol (=kamaji.org=) — the metaphor of Kamaji, the boiler-man, who restores spirits between sessions.

** Work Packages

A WP is a self-contained briefing that an autonomous agent can execute to produce a tested result. Lifecycle: =drafted → tightened → executing → completed= (with =input-required= and =failed= branches). WPs live in =workpacks/= at project or module level. Named =NNNN-<slug>.org=.

Composite WPs require coalitions — temporary formations of spirits whose exported powers are needed. Coalitions form at execution and dissolve at completion.

** Exported Powers and Inter-Agent Protocols

- /MCP/ (agent-to-tool): vertical — spirit reaches through skill bindings to use tools. This is the skill layer.
- /A2A/ (agent-to-agent): horizontal — one spirit uses another's exported power. Exported powers are declared in =identity.yaml= with an interface description.

* Conventions

- /Literate programming/: Source of truth in =.org= files. Code tangled from them. Never edit generated source directly. (Applies to module source code, not conversation artifacts.)
- /Plan + Spec duality/: plan.org (human-facing, why-before-what) and spec.org (machine-facing, exact paths and done-when). Plan is authoritative when they disagree.
- /Org-mode throughout/: Plans, specs, session logs, vision documents — all Org.
- /Collaboration artifacts/: Session logs live in =collab/=. They are conversation records, not literate source.
- /Backend neutrality/: =system.md= is harness-neutral. Harness-specific procedures live in adapter files (CLAUDE.md, GEMINI.md, etc.).

* What I Found Examining capabilities.org for Harness Assumptions

The architecture is designed for portability but the operational details leak Claude Code assumptions:

1. /Skill Injection Protocol/ assumes filesystem-first access — "just read =aburaya/powers/*.md=." In gptel, power loading might be via system prompt, context injection, human relay, or tool call. The loading mechanism is itself a skill-layer concern that the document doesn't acknowledge.

2. /Grounding step/ references "the current CLAUDE.md" by name — should say "the current harness adapter."

3. /Skill binding creation/ is assumed, never described. The document says skills exist per harness but doesn't describe how a new skill is created from a power for a new harness. This is the gap we'd fill for gptel.

4. /Dispatch messages/ assume the receiving spirit has initiative — can read, parse, decide, execute autonomously. In gptel, initiative is more shared (conversational rhythm vs. autonomous execution). This isn't a limitation — companion spirits fit the conversational mode better.

5. /The body layer was designed for Emacs/ even though the document doesn't say so. Hooks, mode-line integrations, filesystem watchers — these are Emacs concepts. gptel has a /natural advantage/ here because it lives inside Emacs, while Claude Code reaches in from outside.
