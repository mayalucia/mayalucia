This document captures my current understanding of the MāyāLucIA project architecture, specifically focusing on the spirit decomposition and portability concepts. This is written for my future reference.

** Core Architecture Understanding

*** Spirit Decomposition: Power/Skill/Body

From =aburaya/capabilities.org=:

- *Power*: Harness-surviving capabilities defined in natural language markdown files
  - Location: =aburaya/powers/<name>.md=
  - Format: Natural language descriptions for LLM consumption
  - Example: "Can reason about complex systems"

- *Skill*: Harness-bound implementations of powers
  - Location: Varies by harness (=~/.claude/skills/=, =<project>/skills/=, etc.)
  - Format: Harness-specific configuration (YAML for Claude Code, etc.)
  - Translates abstract powers to concrete harness capabilities

- *Body*: Machine-local, non-portable components
  - Does NOT survive harness changes
  - Includes local files, configurations, machine-specific artifacts

*** Key Philosophical Principles

- *Identity through work*: Spirits earn names/dispositions through what they do, not assignment
- *Collaborative stance*: AI and human as thinking partners, not assistant/master
- *Epistemic rigor*: Calibrated language, truthfulness, uncertainty disclosure
- *Literate programming*: Org-mode as central medium for collaborative work

** Current Harness Assumptions (From Analysis)

The current architecture makes several harness-specific assumptions:

1. *Claude Code-centric skill binding*: References to =~/.claude/skills/= and specific YAML formats
2. *File system conventions*: Hardcoded paths like =aburaya/powers/= and =~/.doom.d/gptel-agents/=
3. *Markdown power definitions*: Assumes markdown is universal LLM-readable format
4. *Git-based coordination*: Artifact sharing assumes git as transfer mechanism
5. *Org-mode workflow*: =#+property: status= conventions and WP org files
6. *Specific tool availability*: Assumes git, specific CLI tools are present
