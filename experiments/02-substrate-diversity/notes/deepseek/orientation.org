* Overview

MāyāLucIA is a personal computational environment for scientific
understanding through creation. It uses digital twin reconstruction
from sparse data as a vehicle for insight, focusing on two domains:
brain circuits (neuroscience) and mountain valleys (Himalayas). The
project employs a collaborative human-AI approach where both are
complementary intelligences (Sculptor's Paradox). The entire framework
is Org-mode based, with literate programming and git submodules.

* Core Concepts

** Spirits

AI agents with distinct identities and roles (e.g., mayadev, mu2tau
(human), dmt-eval-guardian, sutradhar-guardian). They belong to guilds
(bravli, mayalucia, epistem, apprentis). Spirits are defined in
=aburaya/spirits/<name>/= with an =identity.yaml= that lists powers
they hold and export.

** Power

A natural-language description of a capability, stored in
=aburaya/powers/<name>.md=. Powers are harness-independent: they
describe what the capability does and key principles, but not how to
implement it in a specific environment.

** Skill

The harness-specific binding of a power. For example:
- In Claude Code: a Markdown file in =~/.claude/skills/= or =<project>/skills/=.
- In gptel: a YAML agent definition in =~/.doom.d/gptel-agents/=.
The skill contains concrete details like file paths, tool names, and
harness-specific commands that realise the power.

** Body

Host-specific sensory/motor wiring that enables a spirit to perceive
and act in a particular environment (e.g., Emacs hooks, mode-line
segments). Bodies are only needed for persistent companion spirits;
summoned spirits (short-lived sessions) rely solely on skill tools.

** Sutra Relay

A file-based message bus used for inter-spirit communication and state
announcements. Messages are Markdown files in a =relay/= directory
with a naming convention:
=YYYY-MM-DD-HHMMSS-<host>-<topic>.md=. They carry structured
information (e.g., WP lifecycle transitions, dispatch assignments).
The relay is a project-specific protocol but harness-neutral (any
harness with file I/O can participate).

** Work Packages (WP)

Self-contained tasks defined in Org files, typically located in
=workpacks/= (project-level) or module/domain subdirectories. Naming:
=NNNN-slug.org=. They include context, inventory, specification,
execution order, and acceptance criteria. WPs have a lifecycle with
states: =drafted=, =tightened=, =executing=, =input-required=,
=resumed=, =completed=, =failed=. The current state is stored in the
Org file as =#+property: status= and announced in the relay.

** Composite WP and Coalitions

A composite WP requires capabilities from multiple spirits. It can be
executed in several ways:
- Single agent that loads multiple powers via /skill injection/
  (reading the relevant power definitions and using them as
  instructions).
- Multiple spirits forming a temporary /coalition/, each handling a
  subset, coordinated via the relay.

** Skill Injection

When an agent needs a power it does not natively possess, it reads the
power's Markdown definition and incorporates it into its reasoning.
This is not identity transfer; the agent simply gains the described
capability. Skill injection assumes the agent can read files at
runtime; alternatively, powers can be pre-loaded.

** Dispatch Messages

Structured assignments sent via the relay to delegate work. They
contain:
- DAG patch (dependency graph of WPs)
- Execution order
- Delegation guidance (which guild spirits to involve)
- Host assignment (which machine should run the tasks)
- Parallel work instructions
The receiving spirit owns the decision of how to execute (e.g.,
instantiate guild spirits or do the work itself).

- =aburaya/= -- spirits, powers, guild definitions.
- =agency/= -- AI agent orchestration (likely scripts to launch sessions).
- =collab/= -- collaboration logs and session artifacts.

All modules and domains are git submodules.

* Harness-Bias in capabilities.org

The architecture document =capabilities.org= is largely
harness-neutral but contains a few assumptions reflecting its origin
in Claude Code:

1. *MCP/A2A terminology* -- Uses Claude-specific labels for
   tool-calling (MCP) and inter-agent communication (A2A).
   /Fix/: Replace with generic terms like "tool-calling interface"
   and "inter-agent communication".

2. *Concrete skill-binding examples* -- Gives exact file paths for
   Claude Code and gptel skills. While helpful, they might be taken
   as normative.
   /Fix/: Present as examples in a footnote or callout; emphasise
   that each harness defines its own skill storage mechanism.

3. *Assumption of tool-calling* -- The skill layer description
   implicitly expects the harness to provide a structured
   tool-calling protocol. Some harnesses may lack this.
   /Fix/: Acknowledge alternative implementations: a skill can be a
   set of natural-language instructions that the LLM follows, with
   output parsed by a host-side interpreter.

4. *Skill injection requires runtime file reading* -- Assumes the
   agent can read files on demand. Not all harnesses may support that.
   /Fix/: Note that skill injection can be done either dynamically
   (via file I/O tools) or statically (by including power definitions
   in the initial prompt or WP spec).

5. *Dispatch message format details* -- The specific structure (DAG
   patch, etc.) is a project convention, not an architectural
   necessity. The document could be clearer about this.
   /Fix/: Clarify that the dispatch format is part of the sutra relay
   implementation, not a generic requirement of the capability
   architecture.

6. *Body layer examples are Emacs-centric* -- The examples refer to
   Emacs hooks and mode-line. This is fine as an illustration, but
   the architecture is host-agnostic.
   /Fix/: Add a note that bodies are host-specific and the Emacs case
   is just one possible realisation.

With these adjustments, the document would be fully harness-neutral.
