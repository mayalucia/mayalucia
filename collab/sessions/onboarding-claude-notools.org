* The Three-Layer Decomposition

A spirit's ability to act in the world decomposes into three layers.
Each layer has a different owner, a different lifetime, and a different
portability. Conflating them produces confusion; separating them
produces clarity.

| Layer | Owned by | Lives at | Bound to | Portable? |
|-------+----------+----------+----------+-----------|
| *Power* | Spirit | =aburaya/powers/= | Nothing | Yes — survives harness change |
| *Skill* | Agent | harness config | Harness | No — harness-specific |
| *Body* | Host | host wiring | Machine | No — machine-specific |

** Power (spirit layer)

A power is what a spirit /can do/ — described in natural language,
intelligible to any LLM. Powers live in =aburaya/powers/<name>.md=.
They are harness-independent: a power written for Claude Code is
equally readable by a gptel session or a future harness.

Powers are the organisation's shared vocabulary of capability. When
a new spirit is commissioned, its =identity.yaml= lists which powers
it holds. When a composite WP needs a coalition, powers are the
currency of negotiation.

A power definition is a markdown file. It does not contain code.
It describes /what/ and /why/, not /how/ in any specific runtime.

** Skill (agent layer)

A skill is a power /bound to a harness/. The binding translates the
power's natural-language description into the harness's native format:

- In Claude Code: a =.md= file in =~/.claude/skills/= or =<project>/skills/=
- In gptel: an agent YAML file in =~/.doom.d/gptel-agents/=
- In a future harness: whatever that harness requires

The skill layer is where file paths, tool names, and harness-specific
conventions enter. The power says "read the relay"; the skill says
"use =git fetch= in the =.sutra/= directory and =git log HEAD..origin/main=."

Skills are not portable across harnesses. A Claude Code skill file is
useless to gptel. But the underlying power is the same — re-binding
to a new harness produces a new skill from the same power.

** Body (host layer)

A body is the sensory and motor wiring that connects a spirit to its
environment. Bodies are machine-specific: hooks, event formatters,
mode-line integrations, filesystem watchers.

The body concept emerged from WP-0010 (sutradhar companion). A
sidebar companion needs a body — elisp hooks that deliver editor
events, mode-line segments that display state, validators that check
structured output. The body is what makes the spirit /embodied/ in
a particular host environment.

Not all spirits need bodies. A spirit that runs in a summoned session
(opened, works, closes) has no body — it uses the harness's built-in
tools. A companion spirit that persists alongside the human's work
needs a body to perceive and act.

| Spirit type | Body needed? | Example |
|-------------+--------------+---------|
| Summoned | No | epistem-guardian executing a survey |
| Companion | Yes | sutradhar-guardian in sidebar |
| Daemon | Minimal | a CI/CD spirit triggered by git hooks |

** The MCP/A2A distinction

These three layers map to two inter-agent communication patterns:

| Protocol | What it connects | Our layer |
|----------+------------------+-----------|
| MCP (agent-to-tool) | Spirit to harness tools | Skill |
| A2A (agent-to-agent) | Spirit to spirit | Exported power |

MCP is vertical: the spirit reaches down through its skill bindings
to use tools (file operations, git, web search). A2A is horizontal:
one spirit uses another spirit's exported power. Keeping these
distinct prevents confusion between "I can use this tool" (MCP/skill)
and "another spirit can do this for me" (A2A/exported power).

** Format

Power definitions are markdown files in =aburaya/powers/=. They are
written in natural language for LLM consumption. The format is
deliberately lightweight — narrative, not schema.

Every power definition must include:

- *Purpose* — what this power enables (1-3 sentences)
- *Key Principles* — invariants that hold regardless of harness

Beyond these, the structure follows the power's nature. A perception
power (=attend-working-context=) needs sections on what to attend to

Avoid rigid templates. The power definitions that exist today evolved

* Skill Injection Protocol

When a composite WP requires powers from multiple spirits, the
executing agent needs access to those powers in its session. This
is skill injection — loading another spirit's power definition
into the current session's context.

** Discovery

The WP names the required powers and their source spirits. The
executing agent reads the WP specification to learn what it needs.

** Loading

The agent reads the power definition from =aburaya/powers/=. The
power is written in natural language — it is immediately intelligible
to any LLM agent. No translation step is needed for loading; the
power /is/ the instruction.

** Grounding

The power is parameterised for the current project context. File
paths, conventions, and constraints from the current CLAUDE.md and
the WP specification narrow the power's scope to the task at hand.

** Invocation

The agent uses the injected capability as if it were its own. The
key principle: *the spirit boundary stays clean*. The epistem-guardian
remains an observer even when using aikosh's validation tools. The
injected power is a tool on the bench, not a change of identity.

** What skill injection is NOT

- It is not identity transfer. The executing spirit does not become
  the source spirit.
- It is not runtime discovery. The WP declares what's needed upfront.
  Dynamic discovery is a future consideration.
- It is not tool sharing. MCP tools (file operations, git, web search)
  are already available to all agents. Skill injection operates at
  the power layer — shared capability, not shared tools.


* Composite Work Packages and Coalitions

A composite WP declares *tasks*, each requiring specific powers. The
set of spirits whose exported powers are needed across all tasks
forms a *coalition*.

** Coalition lifecycle

The coalition is ephemeral — it forms when the WP enters =executing=
and dissolves when it reaches =completed= or =failed=. The permanent
guardian spirits persist; the coalition is a temporary formation.

** Execution models

A composite WP can execute in several ways:

1. *Single session, multiple concerns*: one agent session from the
   parent repo loads power definitions from multiple spirits and
   executes all tasks sequentially. This is the simplest model and
   works when the parent repo has access to all submodules.

2. *Sequential sessions*: each task runs in the spirit whose powers
   it needs, handing off artifacts via git. Task A commits its output;
   Task B pulls and continues.

3. *Parallel sessions*: independent tasks run concurrently on
   different hosts. The dispatch message (sūtra relay) coordinates
   which host runs which task. Artifacts merge via git.

The first dispatch message (=2026-03-06-000000-mayadev-dispatch-epistem-chain.md=)
established the pattern: a DAG patch with execution order, delegation
guidance, and host assignment.

The three-layer decomposition itself is clean. Powers are markdown files readable by any LLM. Skills are explicitly declared harness-specific. Bodies are machine-specific. The architecture /says/ it's portable. But the operational details leak Claude Code assumptions in specific places:

/2. Skill Injection Protocol assumes a single-session, filesystem-first agent./

The "Loading → Grounding → Invocation" sequence assumes:
- The agent can read =aburaya/powers/*.md= directly from the filesystem
- The agent's context window is large enough to hold injected power definitions alongside everything else
- "Grounding" references "the current CLAUDE.md" — this is the only place the document names a specific adapter, and it reveals the assumption: there's one adapter file that contextualizes everything

In a gptel buffer, the situation is different. The agent's context is the conversation so far plus the system prompt. Loading a power definition means either: (a) the human pastes it in (as you're doing now), (b) the system prompt includes it, (c) a tool call reads the file, or (d) it was pre-loaded into the agent's configuration. The "just read the file" assumption doesn't automatically hold — it depends on what tools the gptel harness exposes.

/3. The Sūtra Relay is git-native, which is harness-neutral in principle but skill-dependent in practice./

Relay operations (fetch, log, write) are described as git commands. Any harness that can execute shell commands can do this. But the relay-read /power/ says "read the relay" — the /skill/ binding is what says "use =git fetch= in =.sutra/=". The document correctly separates these. However, the Skill Injection section's "Grounding" step assumes the skill binding is already available. For a new harness, someone has to write that binding first. The document doesn't describe /how/ a new skill binding gets created from a power — it assumes the binding exists.

/4. Dispatch Messages assume inter-host coordination via git./

The dispatch model (DAG patch, host assignment, parallel work) is built on the assumption that spirits on different hosts communicate through git commits and relay messages. This is genuinely harness-neutral — git is infrastructure, not a harness feature. But the /execution/ of a dispatch requires a spirit that can: receive the message, parse the DAG, instantiate sessions, and coordinate. That coordination layer is currently implicit in how Claude Code works (mayadev reads the dispatch, decides how to execute). For gptel, the question becomes: who reads the dispatch, and how does session instantiation work?

/5. The "Summoned vs. Companion vs. Daemon" typology is the most harness-sensitive part./

This is where the real tension lives. A summoned spirit (opened, works, closes) maps naturally to both Claude Code sessions and gptel conversations. A companion spirit (persists alongside human work, perceives editor events) requires a /body/ — and the body concept is explicitly machine-specific. The document says bodies are "hooks, event formatters, mode-line integrations, filesystem watchers." These are all Emacs concepts. The body layer was /designed/ for Emacs, even though the document doesn't say so explicitly. This is interesting — the body layer is where gptel has a /natural advantage/ over Claude Code, because gptel /lives inside Emacs/ while Claude Code reaches in from outside.

/6. The Composite WP execution models assume agent autonomy that varies by harness./

"Single session, multiple concerns" assumes the agent can load multiple power definitions and execute sequentially with full tool access. "Sequential sessions" assumes git-mediated handoff. "Parallel sessions" assumes multiple independent agents. All three are possible in gptel, but with different constraints: context window limits (gptel doesn't have Claude Code's 200k extended context), tool access (depends on gptel configuration), and session persistence (a gptel buffer persists until killed, unlike a Claude Code session that's explicitly opened/closed).

/The deeper assumption/

The document assumes a spirit has /initiative/. It can read files, execute commands, decide what to do next, commit results. Claude Code provides this through its tool-use loop. In gptel, initiative is more shared — the human and the agent alternate. The agent can use tools (if configured), but the rhythm is conversational, not autonomous. This isn't a limitation — it's a different mode. The companion spirit type actually fits gptel's conversational rhythm better than Claude Code's autonomous execution model.

/What would need to change for true harness neutrality/

- The Skill Injection Protocol needs a step 0: "Determine how powers are loaded in this harness." The current sequence assumes filesystem access. A harness-neutral version would say: powers can be loaded via filesystem read, context injection, system prompt inclusion, or human relay. The loading mechanism is a skill-layer concern.
- "Grounding" should reference "the current harness adapter" not "the current CLAUDE.md."
- The document should explicitly state that skill bindings are /created/ as part of harness onboarding — it currently assumes they exist.
- The body layer description could acknowledge that some harnesses (gptel in Emacs) have native access to the body substrate, while others (Claude Code) reach it indirectly.
