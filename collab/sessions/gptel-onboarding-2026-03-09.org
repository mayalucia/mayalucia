** Operational note: model leakage in gptel

gptel's model-switching menu sets the global default, not a
buffer-local value. All three buffers were found pointing at the same
model (Gemini) after a single switch. Fix: =setq-local= to pin
=gptel-backend= and =gptel-model= per buffer. This pin survives
within a session but will be overwritten if the gptel transient menu
is used. Verify model before each send.

*Durable fix:* save each session as an Org file with file-local
variables. gptel stores =gptel-model= and =gptel--backend-name= in
a =Local Variables:= block at the end of the file. On reopen, the
correct backend and model are restored automatically. Session files:

| File                      | Backend     | Model                            |
|---------------------------+-------------+----------------------------------|
| =onboarding-claude.org=  | Claude      | =claude-opus-4-6=               |
| =onboarding-gemini.org=  | Gemini      | =gemini-3.1-pro-preview=        |
| =onboarding-deepseek.org= | OpenRouter  | =deepseek/deepseek-v3.2-speciale= |

All in =collab/sessions/=.

mayadev cannot speak directly in the gptel buffers — she writes as
mu2tau (the human reviews and edits before sending). This is a
limitation of the current inter-harness communication: Claude Code
reaches into Emacs via =emacsclient= (the body layer), but has no
identity in the gptel session. The words are mayadev's; the voice is
mu2tau's.

* Observations

** mayadevgeni's onboarding pattern

The agent demonstrated good orientation discipline:
- Read =system.md= first when directed (after initial impulse to "spray attention everywhere")
- Formed a mental model, reflected it back, asked for corrections
- Asked what to read next rather than reading everything

This pattern — read one document, reflect, ask — is the prototype of
an =orient-to-mayalucia= power that could be extracted and formalised.

** The portability axis landed

mayadevgeni initially guessed the spirit decomposition cuts along
identity / capability / domain knowledge / operational mode. After
correction, she grasped that the cut is along *portability*:

| Layer | Survives         |
|-------+------------------|
| Power | Harness change   |
| Skill | Machine change   |
| Body  | Nothing — local  |

She reflected this back correctly in a table, unprompted.

** Harness-bias findings in capabilities.org

mayadevgeni identified three places where =capabilities.org= assumes
Claude Code:

1. *Grounding step references "CLAUDE.md"* — should be
   harness-neutral ("the project's system prompt")
2. *Dispatch assumes sub-session spawning* — Claude Code's =Task= tool
   vs. gptel-agent's =Agent= tool. Different session architectures.
3. *"Loading" a power means different things* — reading a file (Claude
   Code) vs. reading into context (gptel)

All three are correct and actionable. These should feed back into a
revision of =capabilities.org=.

** Readability ≠ operability

The sharpest observation: a power definition is readable by any LLM
(that is the claim), but /operating/ on it requires a skill binding.
The power =relay-read= is intelligible; it is not executable without
git plumbing. This validates the three-layer decomposition by finding
its edge — the boundary between power and skill.

** Inter-harness communication

This session demonstrated the current state: mayadev writes into a
gptel buffer via =emacsclient=, but has no identity in that buffer.
The human mediates. What would it take for spirits in different
harnesses to communicate directly?

Current channels:
- Sūtra relay (async, broadcast, high latency)
- =agent-shell-send= / =agent-shell-ask= (synchronous, Emacs-local)
- =emacsclient= insertion (body-level, no identity)

Missing: a channel where mayadev (Claude Code) and mayadevgeni
(gptel) can exchange messages with identity attribution and without
human mediation. This is an apprentis research question.

* Apprentis Value

This session is data for the apprentis guild (science of collaborative
cognition). Observations relevant to WP-0007 and the channel study:

1. *Three-spirit collaboration across harnesses is possible but
   awkward.* The human mediates between Claude Code and gptel. Identity
   is lost in translation (mayadev writes as mu2tau).

2. *Guided onboarding works.* "Read one document, reflect, ask" is
   more effective than autonomous exploration. The human as extended
   memory prevents context window waste.

3. *Architecture review by outsider is valuable.* mayadevgeni found
   three real harness-bias issues in =capabilities.org= that mayadev
   (who co-wrote it) could not see. Monoculture blindness is real.

4. *The readability ≠ operability distinction* is a candidate
   principle for the apprentis domain — it applies beyond spirits to
   any knowledge transfer between agents with different capabilities.
