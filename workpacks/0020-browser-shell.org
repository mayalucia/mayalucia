* Specification

** What to build

Two views sharing one data source and one info panel:

*** 1. Geography view

A 2D SVG map of the Western Himalaya. Stories rendered as pins at their
geographic locations. Requirements:

- Regions positioned by relative geography (Tirthan south-east, Lahaul
  north, Parvati east, Baspa/Kinnaur east, Sutlej south, Karakoram
  far north)
- Tirthan cluster (stories 7, 10, 11, 12) visually separable — zoom
  or spatial spread by altitude
- Karakoram region present but empty (future territory)
- Story #6 (Phantom Faculty, abstract) positioned outside the map or
  in a special "abstract" zone
- Click a pin → info panel shows story metadata
- The Thread Walker's route as a connecting path between stories she
  appears in

For Phase 1, the map is *schematic* — positioned SVG elements, not
a DEM-derived topographic rendering. Correct relative positions matter;
cartographic accuracy is Phase 2+.

*** 2. Concept view

A force-directed graph of translation keys. Fork from the constellation
viewer's D3 simulation.

- Concept nodes — the 94 concepts from =stories.edn=
- Story-to-concept edges — each story connects to its concept list
- Concept lineages — highlighted when a concept appears across multiple
  stories (from =:edges= with =:type :concept-lineage=)
- Concept bridges — related concepts that span stories (from =:edges=
  with =:type :concept=)
- Hover a concept → highlight all stories that translate it
- Click a story → info panel

Grouping by =:introduced-in= story number provides visual clustering.

*** 3. Info panel

Shared between both views. On story selection, shows:

- Title and story number
- Setting: region, places, altitude, season
- Document type
- Illustration thumbnails (paths from =:image-dir= and =:illustrations=)
- Characters (names and roles)
- Translation keys (concept name + story equivalent)
- Description (excerpt)
- Link to full story on mayalucia.dev

*** 4. View switching

Toggle between geography and concept projections. Same data, different
spatial layout. The info panel persists across view switches if a story
is selected. URL fragments for deep-linking: =#geography=, =#concept=,
=#story/thread-walkers=.

** Design constraints

- The browser is a *navigation device*, not a reading surface. Clicking
  a story leads to the Hugo page, not inline text.
- Phase 1 is two views. Additional views (time, document type, voice,
  emergent dimensions) are Phase 2+ and depend on WP-0014 human
  annotations.
- The schematic map in Phase 1 should be designed so a DEM-derived
  topographic map is an upgrade, not a rewrite. SVG layers, not
  hardcoded positions.
- Mobile is not a priority for Phase 1. Desktop-first.
- Dev server port: 3335 (constellation viewer uses 3334).

* Execution Order

1. *Setup* — create =website/story-browser/= with deps.edn, shadow-cljs.edn,
   package.json. Verify shadow-cljs builds an empty app.
2. *Data* — copy =workpacks/data/stories.edn= into =data.cljs= (or load
   as EDN at build time). Wire into Reagent state.
3. *Concept view* — fork force simulation from constellation viewer.
   Render concepts as nodes, stories as connecting elements. Info panel
   on selection.

   pins. Tirthan cluster separable. Karakoram empty.
5. *View switching* — toggle between geography and concept views.
   Shared info panel.
6. *Hugo integration* — create =website/content/writing/browser.md=
   page. Wire JS output.
7. *Polish* — node styling, edge types, hover states, transitions.

Steps 3 and 4 are independent and could run in parallel if the executor
has sufficient context. Step 5 integrates them.

* What This Proves

That the constellation viewer generalises. The same ClojureScript +
D3-force + Reagent stack that renders the project's own structure
can render the project's creative output. The viewer is a genus, not
a species. And the EDN data format — the same format that describes
project entities — describes stories, their concepts, their
characters, and the edges between them.

* Open Questions

- *Node aesthetics*: what do story nodes look like? The constellation
  viewer uses "phase crystals." The story browser needs something
  that evokes the document type — field notes, logbook, survey,
  testimony, satire. Placeholder circles are fine for Phase 1.
- *Map base layer*: should the schematic map have elevation contours,
  river lines, or just region labels? Phase 1 can start minimal.
- *Story #10h (Hindi)*: appears as a variant within story #10's EDN
  entry. Does it get its own pin or appear in the info panel?
- *Illustration paths*: thumbnails need to resolve to
  =website/static/images/writing/<image-dir>/<filename>=. The data
  has =:image-dir= and =:illustrations= — verify path construction
  works in the deployed Hugo context.
