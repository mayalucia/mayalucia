* Context and Motivation

** The gazetteer need

WP-0003 produces a comprehensive Org document on the spirit ecology of
the transborder Himalaya. But a comprehensive document is the wrong
format for an agent that needs to compose a story set in Ladakh at 3am.
The agent does not need the entire survey — it needs: "what spirits live
near Pangong Tso? what do the locals call a water spirit? what is the
possession protocol in this region?"

Currently, =aburaya/powers/story-compose.md= carries a single
translation table (Miyazaki → Himalaya) and a geography table (stories
→ settings). These are flat markdown tables loaded into context every
time. This works at the Kullu scale (~10 entries). It will not work at
the transborder Himalaya scale (~100+ entries across 8 regions).

The problem is token economics. A 50-page knowledge base loaded into
every story-compose invocation wastes context window on material that is
99% irrelevant to the specific story being written. What we need is a
structure the agent can /query/ — pull in only the entries relevant to
the story's setting, altitude zone, and thematic needs.

** The capability architecture need

This WP is also the first *composite work package* — multiple tasks
requiring different spiritual competencies. Populating the gazetteer
needs the epistem-guardian's observer eye. Validating entries needs
aikosh-guardian's builder discipline. Documenting the capability
architecture needs the sutradhar's coherence concern.

Rather than designing a cross-spirit protocol in the abstract (the
former WP-0005), we define it /here/, in the WP that first needs it.
The protocol emerges from doing, not from planning.

*** The coalition model

A composite WP declares *tasks*, each with its own skill needs. Skills
map to spirits. The execution spawns what's needed — ephemeral
coalitions of spirits contributing their powers to a shared endeavour,
each retaining their identity.

This is different from simple skill injection (one spirit borrows
another's tool). A coalition is a *formation* — multiple spirits with
shared state (the WP's artifacts in git), each contributing their
domain, dissolving when the WP completes.

** Lessons from the A2A Protocol

Google's Agent2Agent (A2A) protocol (2025, now under the Linux
Foundation) solves inter-agent communication across /organisations/
— different companies, different frameworks, different servers. Our
problem is inter-spirit collaboration within /one organisation/ —
different concerns, same codebase, same relay. But the conceptual
architecture is instructive.

*** What A2A gets right

| A2A Concept | Our Equivalent | Gap |
|-------------+----------------+-----|
| *Agent Card*: JSON declaring identity, capabilities, endpoint | =identity.yaml=: five-layer ontology with powers | Our cards don't declare /exported/ powers — what the spirit offers others |
| *Task lifecycle*: submitted → working → input-required → completed/failed | WP execution: no formal state machine | WPs need lifecycle states, transitions via sūtra |
| *Opacity*: agents collaborate without exposing internal state | "The harness is not the spirit" | Already a principle; coalition must preserve it |
| *Message Parts*: TextPart, FilePart, DataPart | Sūtra relay: markdown + YAML frontmatter | Relay lacks structured artifact exchange |
| *MCP vs A2A*: agent-to-tool vs agent-to-agent | Skill (tool-binding) vs Power (spirit-level) | We've been conflating these two layers |

*** What we take

1. *Exported powers in =identity.yaml=*: an =exports:= section
   listing powers this spirit makes available to other spirits. Not
   all powers are exported; some are internal. This is the Agent
   Card's =skills= array adapted for our ontology.

2. *Task state machine for WP execution*: =drafted → tightened →
   executing → input-required → completed / failed=. State
   transitions announced in the sūtra.

3. *The MCP/A2A distinction mapped to our layers*: MCP is
   agent-to-tool (our skill layer — power bound to harness). A2A is
   agent-to-agent (our cross-spirit layer — one spirit using another
   spirit's exported power). Keeping these distinct prevents
   conflation.

*** What we don't take

- *HTTP/JSON-RPC transport* — our spirits communicate through the
  sūtra (git-based, append-only) and shared filesystem.
- *Streaming/SSE/webhooks* — our spirits execute sequentially, one
  session at a time. Artifacts handed off via git, not real-time
  streams.
- *Enterprise auth* — spirits trust each other within the
  organisation. Git permissions suffice.

** Task A: The Gazetteer

*** Entry structure

The gazetteer uses a *hybrid* layout. Genus-level files are the
canonical entries — self-contained, story-compose reads these directly.
Species and instance files are *projections* — derived or hand-written
for entries that need individual identity (GIS coordinates, digital
twin queries).

Each spirit YAML file represents a *genus* (type of being) in a
specific regional tradition. The schema needs a hierarchical
classification: genus → species → named instance. It also needs an
expanded landscape vocabulary covering the full range of mountain
terrain (peaks, ridges, passes, gorges, glaciers, moraines, sacred
groves, trade routes — not just valley-floor features).

*Design decision*: both. Genus files are canonical and self-contained
(approach A). Species and instance files are projections (approach B)
— derived from the genus file or hand-written when an entry needs its
own identity (GIS coordinates, detailed narrative). Story-compose reads
genus files. The digital twin queries species and instance files.

*** Why YAML, not Org

- *Queryable*: an agent can glob for =region: ladakh= or
  =altitude_zone: glacial= with grep, without loading all entries
- *Composable*: story-compose loads only the entries matching the
  story's setting
- *Parseable*: Clojure (sutradhar), Python (parbati tools), and
  any LLM agent can read YAML natively
- *Git-friendly*: one file per entry, clean diffs
- *Not too expensive*: a genus file is ~60 lines / ~800 tokens.
  Story-compose loads 2-4 genus files per story (~2000-3000 tokens),
  not the 50,000 of a full knowledge base. Species and instance
  files are ~20 lines / ~300 tokens each — loaded only when the
  digital twin or a story needs fine-grained detail.

Org remains the source of truth for the /survey/ (WP-0003). YAML is
the /queryable extract/ — the gazetteer is derived from the survey,
not a replacement for it.

*** Query protocol for story-compose

When composing a story, the agent:

1. Determines the story's geographic setting (region, valley, altitude)
2. Queries the gazetteer:
   - =grep -rl "region: <region>" domains/epistem/collections/himalayan-spirits/spirits/genera/=
   - =grep -rl "altitude_zone: <zone>" domains/epistem/collections/himalayan-spirits/spirits/genera/=
3. Reads the matching entries (typically 3-8 files, ~2000-4000 tokens)
4. Uses the =vocabulary:= fields for story language
5. Uses the =nature:= and =diagnostic:= fields for thematic depth

This replaces the single flat translation table with a geographic lookup. The system → story table in =story-compose.md= remains (it maps system concepts to story register). The Miyazaki table either becomes region-specific entries in the gazetteer or is kept as a compact cultural-bridge table for readers who know the anime.

*** Translation table evolution

The current monolithic tables in =story-compose.md= evolve:

| Current | After gazetteer |
|---------+-----------------|
| Miyazaki → Himalaya (monolithic) | Compact cultural-bridge table (kept small, for humans, not for agent queries) |
| System → Story (monolithic) | Kept as-is — this is structural, not geographic |
| No geographic lookup | Gazetteer query: region + altitude → vocabulary |

The power definition (=story-compose.md=) shrinks. It says: "query the gazetteer at =domains/epistem/collections/himalayan-spirits/= for your setting's spirit vocabulary." The knowledge moves out of the power definition and into queryable data.

** Task B: Capability Architecture

The three-layer decomposition, the skill injection protocol, and the
coalition model — all defined here because this WP is where they are
first /used/.

*** B.1 The three-layer decomposition

| Layer | Owned by | Lives at | Bound to | Example |
|-------+----------+----------+----------+---------|
| Power | Spirit | =aburaya/powers/= | Nothing (portable) | =story-compose.md= |
| Skill | Agent | =~/.claude/skills/= | Harness | bedtime-story (Claude Code) |
| Grounding | Project | =<project>/skills/= | Project | sutradhar story-compose |

*** B.2 Power authorship protocol

- Who can author a power? (The spirit whose work produces the
  capability, or any spirit that identifies a reusable pattern)
- What format? (Define required sections for a power definition)
- Where do they live? (=aburaya/powers/= for organisation-wide;
  =<module>/powers/= for module-specific?)
- How are they versioned? (Git-tracked, same as everything else)

*** B.3 Skill derivation

How a power becomes a skill:
- The harness reads the power definition
- The skill file translates it into harness-specific instructions
  (for Claude Code: a =.md= file in =~/.claude/skills/= or
  =<project>/skills/=)
- The skill may add harness-specific details (tool names, file
  paths, Claude Code conventions)

*** B.4 Skill injection protocol

How one spirit's power is made available in another spirit's session:

- *Discovery*: the executing agent reads the WP, which names the
  required powers and their source spirits
- *Loading*: the agent's session loads the power definition from
  =aburaya/powers/= (or the source module's powers directory)
- *Grounding*: the power is parameterised for the current project
  context (file paths, conventions, constraints)
- *Invocation*: the agent uses the injected capability as if it
  were its own skill

The key principle: *the spirit boundary stays clean*. The
epistem-guardian remains an observer even when using aikosh's
validation tools. The injected skill is a tool on the bench, not a
change of identity.

*** B.5 Exported powers (Agent Card pattern)

Each spirit's =identity.yaml= gains an =exports:= section declaring
which powers this spirit makes available to other spirits:

#+begin_src yaml
character:
  powers: [relay-read, validate-collection, query-collection]
  exports:                           # what other spirits can import
    - power: validate-collection
      definition: aburaya/powers/validate-collection.md
      interface: >
        Takes a collection directory path, validates all entries
        against the schema, reports inconsistencies.
    - power: query-collection
      definition: aburaya/powers/query-collection.md
      interface: >
        Takes region, altitude_zone, landscape filters. Returns
        matching genus file paths.
#+end_src

Not all powers are exported. =relay-read= is a personal power —
every spirit uses it for itself, not for others. Exported powers
are the ones a spirit offers as a service.

*** B.6 WP execution lifecycle

WPs have a formal lifecycle. State transitions are announced in
the sūtra:

#+begin_example
drafted → tightened → executing → completed
                   ↘              ↗
               input-required → resumed
                                  ↘
                               failed
#+end_example

| State | Meaning | Transition by |
|-------+---------+---------------|
| =drafted= | WP written, not yet reviewed | Author (spirit or human) |
| =tightened= | Spec reviewed and approved | Human |
| =executing= | Agent working on the WP | Executing spirit |
| =input-required= | Agent blocked, needs human input | Executing spirit |
| =resumed= | Human provided input, work continues | Human |
| =completed= | Acceptance criteria met | Executing spirit + human |
| =failed= | Cannot complete, reasons documented | Executing spirit |

The sūtra relay carries lifecycle messages:

#+begin_example
relay/2026-03-05-140000-macbook-wp0004-executing.md
---
from: macbook-dropbox/opus
date: 2026-03-05T14:00:00
tags: [epistem, wp-lifecycle]
---
WP-0004 (spirit gazetteer + capability architecture) → executing.
Coalition: epistem-guardian, aikosh-guardian. Task A this session.
#+end_example

*** B.7 The MCP/A2A distinction in our architecture

| Protocol | A2A term | Our term | What it connects |
|----------+----------+----------+------------------|
| MCP | Agent-to-tool | Skill | Spirit ↔ harness tools (file ops, git, web) |
| A2A | Agent-to-agent | Exported power | Spirit ↔ spirit (via skill injection) |
| — | — | Task grounding | Skill ↔ project (concrete paths, conventions) |

The skill injection protocol (§B.4) is our A2A. The skill derivation
protocol (§B.3) is our MCP binding. Keeping these layers distinct
prevents a spirit from confusing "I can use this tool" (MCP/skill)
with "another spirit can do this for me" (A2A/exported power).

*** B.8 Composite work packages and coalitions

A composite WP consists of *tasks*. Each task declares:
- What it does (acceptance criterion)
- What skills it requires (mapped to powers)
- Which spirit holds those powers (by =exports:= reference)

The coalition is the set of spirits whose exported powers are needed
across all tasks. The coalition is ephemeral — it forms when the WP
enters =executing= and dissolves when it reaches =completed= or
=failed=. The permanent guardian spirits persist; the coalition is a
temporary formation.

How a composite WP executes:
- A single agent session can serve multiple spirits if it has access
  to all their repos (e.g. parent repo with submodules).
- Or sequential sessions: each task runs in the spirit whose powers
  it needs, handing off artifacts via git.
- The WP's lifecycle state is tracked via sūtra relay messages.
- Each task within the composite can have its own sub-state, but the
  WP-level state governs the sūtra announcements.

*** B.9 Power format specification

Standard structure for power definition files:

#+begin_example

  ## Protocol       — step-by-step procedure

  ## Authored by    — which spirit developed it
  ## Used by        — which spirits currently use it
#+end_example

* Resolved Questions

- *Granularity*: hybrid. Genus files (=genera/=) are canonical and
  self-contained. Species (=species/=) and instance (=instances/=)
  files are projections for fine-grained queries and the digital twin.
  Resolved during WP-0004 spec-tightening.

- *Who maintains the gazetteer?* The epistem guild (WP-0001). The
  aikosh-guardian holds the tooling; the gazetteer is epistem's first
  collection. The sutradhar uses it (for stories), parbati uses it
  (for GIS), aburaya consults it (for naming).

- *Where does it live?* =domains/epistem/collections/himalayan-spirits/=
  — not under parbati (too narrow) or aburaya (wrong concern).
  Epistem is the domain of structured knowledge for agents.

- *Capability architecture location*: not a separate WP. Designed
  here, in the WP that first needs it. The Feynman principle: the
  protocol emerges from doing, not from planning in the abstract.
  (Former WP-0005, merged.)

* Open Questions

- *Versioning and authority*: the survey (Org, WP-0003) is the
  authority. The gazetteer (YAML) is derived. If they disagree, the
  survey wins. Should we tangle the YAML from the Org? Possible but
  may be over-engineering at this stage.

- *Raw knowledge base*: the web research backing the extracts needs
  a home outside git — a database, not a repository. This is a future
  aikosh work package (noted in WP-0001).

- *Module-level powers*: should modules define their own powers
  (e.g. =modules/aikosh/powers/validate-collection.md=) or should
  all powers live in =aburaya/powers/=? The former keeps powers
  close to the code; the latter keeps the registry centralised.

- *Harness-agnostic skill injection*: the protocol should work for
  Claude Code, gptel, and future harnesses. But each harness loads
  skills differently. Should the protocol define an abstract
  interface, or document each harness separately?

- *Power composition*: can powers compose? (e.g. =review-compose=
  = =web-survey= + =summarize-corpus= + =org-content-workflow=).
  If so, how is composition expressed in the power definition?

- *Runtime vs static injection*: should skill injection happen at
  session start (static — the CLAUDE.md or WP lists what to load)
  or during execution (dynamic — the agent discovers it needs a
  power and loads it mid-session)?

- *Artifact exchange*: A2A's Message Parts (TextPart, FilePart,
  DataPart) support structured artifact exchange between agents. Our
  relay uses markdown + YAML frontmatter. As composite WPs become
  common, will we need richer artifact handoff in the sūtra — or does
  git (shared filesystem) already provide this?

- *Coalition lifecycle*: should the sūtra track coalition formation
  and dissolution explicitly, or is the WP lifecycle sufficient?
  If coalitions become common, tracking them may reveal patterns
  about which spirits collaborate frequently.
