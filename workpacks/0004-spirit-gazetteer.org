* Context and Motivation

WP-0003 produces a comprehensive Org document on the spirit ecology of
the transborder Himalaya. But a comprehensive document is the wrong
format for an agent that needs to compose a story set in Ladakh at 3am.
The agent does not need the entire survey — it needs: "what spirits live
near Pangong Tso? what do the locals call a water spirit? what is the
possession protocol in this region?"

Currently, =aburaya/powers/story-compose.md= carries a single
translation table (Miyazaki → Himalaya) and a geography table (stories
→ settings). These are flat markdown tables loaded into context every
time. This works at the Kullu scale (~10 entries). It will not work at
the transborder Himalaya scale (~100+ entries across 8 regions).

The problem is token economics. A 50-page knowledge base loaded into
every story-compose invocation wastes context window on material that is
99% irrelevant to the specific story being written. What we need is a
structure the agent can /query/ — pull in only the entries relevant to
the story's setting, altitude zone, and thematic needs.

* Inventory (What Already Exists)

** Translation tables in story-compose.md

Two flat tables:

1. *System → Story* (10 entries): spirit registry → brass-plate ledger,
   relay → wind/water/thread, harness → visit/inhabitation, etc.
2. *Miyazaki → Himalaya* (10 entries): kami → devta, onsen → kund,
   kamikakushi → devta ka aavesh, etc.

These are the prototype. They work but they conflate two concerns:
- /Vocabulary mapping/ (what word to use in a story)
- /Geographic grounding/ (where the vocabulary applies)

The Miyazaki table is actually Miyazaki → Kullu. A story set in
Ladakh needs Miyazaki → Ladakh (different vocabulary: lha not devta,
klu not nāg). A story set in the Pamirs needs peri not devta. The
single table cannot serve multiple regions.

** Existing geographic reference

The story-compose power lists 5 geographic zones with real places. The published stories span: Parvati, Tirthan, Baspa, Spiti/Lahaul, and the Kullu-Tibet border. The Thread Walker moves between them.

** Existing technology

- Org-mode: the human's native format, handles tables and properties
- YAML: used for identity descriptors, guild files, relay frontmatter
- EDN: used for constellation data (Clojure)
- GeoJSON: standard for spatial data (parbati domain)
- Org-mode properties and drawers: per-heading metadata, searchable

* Specification (What You Build)

** The gazetteer format

A *spirit gazetteer*: a structured, queryable collection of entries about supernatural beings, practices, and places across the transborder Himalaya.

*** Entry structure

The gazetteer uses a *hybrid* layout. Genus-level files are the
canonical entries — self-contained, story-compose reads these directly.
Species and instance files are *projections* — derived or hand-written
for entries that need individual identity (GIS coordinates, digital
twin queries).

#+begin_example
domains/epistem/collections/himalayan-spirits/

      kullu-gur.yaml
      ladakh-lha.yaml
      ladakh-klu.yaml
      tibet-yullha.yaml
      tibet-btsan.yaml
      tibet-sabdag.yaml
      kinnaur-nag.yaml
      pamir-peri.yaml
      nepal-jhakri.yaml
      ...
    species/                        # projections: one file per subtype
      kullu-nag-devta.yaml          #   back-references genus
      kullu-ban-devta.yaml
      kullu-rath-devta.yaml
      ladakh-yul-lha.yaml

Each spirit YAML file represents a *genus* (type of being) in a
specific regional tradition. The schema needs a hierarchical
classification: genus → species → named instance. It also needs an
expanded landscape vocabulary covering the full range of mountain
terrain (peaks, ridges, passes, gorges, glaciers, moraines, sacred
groves, trade routes — not just valley-floor features).

*Design decision*: both. Genus files are canonical and self-contained
(approach A). Species and instance files are projections (approach B)
— derived from the genus file or hand-written when an entry needs its
own identity (GIS coordinates, detailed narrative). Story-compose reads
genus files. The digital twin queries species and instance files.

*** Landscape primitives

Available values for the =landscape= field. Each maps to a GIS
primitive (point, line, polygon) for the digital twin.

| Category       | Values                                                       |
|----------------+--------------------------------------------------------------|
| Settlements    | village, town, monastery, hermitage                          |
| Structures     | temple, shrine, chorten, bridge                              |
| Water          | river, stream, spring, lake, hot-spring, glacier, waterfall  |
| Terrain        | peak, ridge, col/pass, gorge, cliff, cave, moraine, scree   |
| Vegetation     | forest, alpine-meadow, sacred-grove, field/terrace           |
| Routes         | trade-route, pilgrimage-path, pastoral-track                 |

*** Genus file (canonical) — =genera/kullu-devta.yaml=

#+begin_src yaml
genus: devta
region: kullu
tradition: hindu-pahari
altitude_zone: [valley-floor, alpine]
landscape: [village, temple, forest, river, peak]    # genus-level defaults

nature: >
  Territorial village god. Each devta governs a jurisdiction roughly
  matching a watershed or administrative unit. Speaks through a gur
  (oracle-medium) during trance possession. Travels in a rath (chariot)
  during festivals. Makes decisions about community affairs.

# story-compose vocabulary (genus-level)
vocabulary:
  spirit_type: devta
  medium: gur
  possession: aavesh
  shrine: than
  festival: jatra
  chariot: rath
  jurisdiction: ilaaqa

# subtypes known in this region
species:
  - name: nāg devta
    gloss: serpent-god of water bodies
    landscape: [spring, lake, hot-spring, river, gorge, waterfall]
    vocabulary:
      water_body: kund
      prohibition: do not pollute or divert
  - name: ban devta
    gloss: forest deity
    landscape: [forest, sacred-grove, ridge]
  - name: rath devta
    gloss: chariot-travelling village god
    landscape: [village, temple, field/terrace, trade-route]

# named spirits (optional — for entries with enough documentation)
notable_instances:
  - name: Hadimba Devi
    species: ban devta
    place: Manali (Dhungri temple)
  - name: Jamlu Devta
    species: rath devta
    place: Malana
  - name: Buddhi Nagin
    species: nāg devta
    place: Naggar

# functional mapping to aburaya
aburaya_analogue: guardian
aburaya_notes: >
  Territorial, bound to a place. The devta does not wander. The closest
  system concept is the guardian spirit (bound to a module/domain).

# what this tradition measures
diagnostic: >
  Community disputes, agricultural timing, illness attribution, resource
  allocation. The devta's pronouncements through the gur encode collective
  judgment about matters too complex for individual decision.

sources:
  - "Chetwode, Kulu: The End of the Habitable World, 1972"
  - "Rosser, fieldwork in Kullu villages, 1950s"
#+end_src

*** Why YAML, not Org

- *Queryable*: an agent can glob for =region: ladakh= or
  =altitude_zone: glacial= with grep, without loading all entries
- *Composable*: story-compose loads only the entries matching the
  story's setting
- *Parseable*: Clojure (sutradhar), Python (parbati tools), and
  any LLM agent can read YAML natively
- *Git-friendly*: one file per entry, clean diffs
- *Not too expensive*: a genus file is ~60 lines / ~800 tokens.
  Story-compose loads 2-4 genus files per story (~2000-3000 tokens),
  not the 50,000 of a full knowledge base. Species and instance
  files are ~20 lines / ~300 tokens each — loaded only when the
  digital twin or a story needs fine-grained detail.

Org remains the source of truth for the /survey/ (WP-0003). YAML is
the /queryable extract/ — the gazetteer is derived from the survey,
not a replacement for it.

** Query protocol for story-compose

When composing a story, the agent:

1. Determines the story's geographic setting (region, valley, altitude)
2. Queries the gazetteer:
   - =grep -rl "region: <region>" domains/epistem/collections/himalayan-spirits/spirits/genera/=
   - =grep -rl "altitude_zone: <zone>" domains/epistem/collections/himalayan-spirits/spirits/genera/=
3. Reads the matching entries (typically 3-8 files, ~2000-4000 tokens)
4. Uses the =vocabulary:= fields for story language
5. Uses the =nature:= and =diagnostic:= fields for thematic depth

This replaces the single flat translation table with a geographic lookup. The system → story table in =story-compose.md= remains (it maps system concepts to story register). The Miyazaki table either becomes region-specific entries in the gazetteer or is kept as a compact cultural-bridge table for readers who know the anime.

** Translation table evolution

The current monolithic tables in =story-compose.md= evolve:

| Current | After gazetteer |
|---------+-----------------|
| Miyazaki → Himalaya (monolithic) | Compact cultural-bridge table (kept small, for humans, not for agent queries) |
| System → Story (monolithic) | Kept as-is — this is structural, not geographic |
| No geographic lookup | Gazetteer query: region + altitude → vocabulary |

The power definition (=story-compose.md=) shrinks. It says: "query the gazetteer at =domains/epistem/collections/himalayan-spirits/= for your setting's spirit vocabulary." The knowledge moves out of the power definition and into queryable data.

** GIS integration (future)

Each =places/= entry can include coordinates:

#+begin_src yaml
name: Manikaran Kund
region: kullu
type: hot-spring
coordinates: [31.9985, 77.3467]
spirits: [nag-devta, shiva]
#+end_src

This makes the gazetteer a bridge between the spirit layer and the
GIS digital twin. A future parbati visualisation could overlay spirit
jurisdictions on the DEM. The YAML is convertible to GeoJSON with a
trivial script.

* Execution

** Who runs this WP

This is the first dual-spirit work package. Populating the gazetteer
requires the epistem-guardian's judgment (what enters the collection,
faithful rendering of traditions) and the aikosh-guardian's discipline
(schema compliance, back-reference integrity, controlled vocabulary).

| Phase | Spirit | Action |
|-------+--------+--------|
| Extraction | epistem-guardian | Read WP-0003 survey, select entries, write narrative fields (nature, diagnostic, sources) |
| Validation | aikosh-guardian | Validate YAML against schema, check back-references, verify landscape vocabulary, confirm token budget |
| Review | epistem-guardian | Review validated entries for faithfulness to source |

Practically: a single agent session from the parent repo (which has
access to both =domains/epistem/= and =modules/aikosh/= as submodules)
can invoke both spirits' concerns. Or two sequential sessions — one
to populate, one to validate.

** Order

1. *Design* (this WP) — define the entry schema, directory structure,
   query protocol. ← done (spec-tightened)
2. *Seed from existing* — extract the ~10 entries implied by the
   current story-compose tables and published stories into YAML files.
   This can happen now, before WP-0003.
3. *Populate from WP-0003* — when the spirit ecology survey is

   (aikosh-guardian).

   to the gazetteer + the compact cultural-bridge table.
6. *Test* — compose a story set in a non-Kullu region (e.g. Ladakh)
   using the gazetteer query protocol. Verify that the vocabulary is
   correct and the context cost is manageable.

Steps 1-2 can proceed immediately. Steps 3-6 depend on WP-0003.

* Resolved Questions

- *Granularity*: hybrid. Genus files (=genera/=) are canonical and
  self-contained. Species (=species/=) and instance (=instances/=)
  files are projections for fine-grained queries and the digital twin.
  Resolved during WP-0004 spec-tightening.

- *Who maintains the gazetteer?* The epistem guild (WP-0001). The
  aikosh-guardian holds the tooling; the gazetteer is epistem's first
  collection. The sutradhar uses it (for stories), parbati uses it
  (for GIS), aburaya consults it (for naming).

- *Where does it live?* =domains/epistem/collections/himalayan-spirits/=
  — not under parbati (too narrow) or aburaya (wrong concern).
  Epistem is the domain of structured knowledge for agents.

* Open Questions

- *Versioning and authority*: the survey (Org, WP-0003) is the
  authority. The gazetteer (YAML) is derived. If they disagree, the
  survey wins. Should we tangle the YAML from the Org? Possible but
  may be over-engineering at this stage.

- *Raw knowledge base*: the web research backing the extracts needs
  a home outside git — a database, not a repository. This is a future
  aikosh work package (noted in WP-0001).
