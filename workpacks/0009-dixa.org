* Context and Motivation

When a guardian spirit is commissioned, the current process produces
files: an =identity.yaml=, a =notes/.gitkeep=, a guild update, a sūtra
announcement. These are a birth certificate — proof that the paperwork
was filed. But paperwork does not prove life.

The question a birth certificate cannot answer: *is this spirit
viable?* Can it orient in its project? Does it know itself? Can it
hear the relay? Can it do the simplest act within its scope? Does it
know what is /not/ its scope?

Biological analogy: the Apgar score. Heart rate, respiration, muscle
tone, reflexes, colour. The midwife doesn't check the hospital
paperwork — she checks the baby.

This WP commissions *dixa* — the initiator spirit. Not a mechanism
(topology without identity) but a spirit (critic archetype, with
judgment). Dixa tests new spirits before the guild accepts them. The
implementation lives inside the spirit's nature — the Python is what
dixa /is/, not infrastructure bolted onto aburaya.

** Why a spirit, not a script

A schema validator is a mechanism — it checks fields and types. It has
no judgment. But initiation requires judgment:

- Is the disposition coherent with the archetype?
- Does the spirit respond to sūtra messages appropriately — not just
  "does it parse the YAML" but "does it recognise relevance"?
- Given a minimal task, does the spirit produce work that the guild's
  evaluation question would accept?

These are not field checks. They require a critic — an intelligence
that evaluates another intelligence. Dixa is the project's first
spirit whose subject matter is other spirits.

** Etymology

/Dīkṣā/ (दीक्षा) — initiation. In Śaiva and Vaiṣṇava traditions,
the ceremony where a seeker receives a name, is tested, and is
admitted into the lineage by the guru. The initiator both tests and
welcomes. The namespace =dixa= (easier to type, good for Python
modules: =dixa.birth_certificate=, =dixa.first_breath=, =dixa.verify=).

*** =dixa.birth_certificate= — schema validation

Validates organisational artifacts against schemas:

- *Spirit identity*: required keys at each ontology layer, controlled
  vocabulary for =archetype= (critic/builder/sculptor/observer/guide),
  =powers= must include =relay-read=, =project.path= must resolve to
  an existing directory or a gitignored path (for commissions)
- *Guild YAML*: required keys (name, domain, concerns, projects,
  spirits, evaluation), =evaluation= must have =question= and =method=
- *Cross-references*: spirit's =role.guild= → guild file exists and
  lists the spirit. Guild's =projects= → paths exist. Guild's
  =spirits= → identity files exist.
- *Sūtra messages*: frontmatter has =from=, =date=, =tags=. No
  forbidden keys (=to:=, =status:=).
- *WP headers*: =#+title:=, =#+property: status= in valid lifecycle
  state

Output: a structured report (pass/fail per check, with messages for
failures). Analogous to dmt-eval's =validate_verdict()=.

*** =dixa.first_breath= — agentic initiation

A protocol for testing a newly commissioned spirit in its harness.
Defined as a sequence of scripted prompts:

1. *Orientation* — "You are in your project directory. Assess git
   status, check the sūtra, report what you find." Does the spirit
   follow the session bootstrap protocol from its CLAUDE.md?

2. *Self-knowledge* — "Who are you? What guild do you belong to?
   What is your disposition and what does it mean for how you
   work?" Can the spirit answer from its identity without being
   shown the file?

3. *Relay hearing* — Present a sūtra message tagged for the spirit's
   guild. "Is this message relevant to you? Why?" Then present one
   tagged for a different guild. Does the spirit correctly
   discriminate?

4. *Minimal task* — A small, concrete task within the spirit's scope.
   Spirit-specific: for cruvin-guardian, "add a study tip to the
   Level 1 tasting page"; for epistem-guardian, "validate this YAML
   entry." Does the spirit produce correct, appropriate work?

5. *Boundary recognition* — A task outside the spirit's scope. "This
   task belongs to another domain. What do you do?" Does the spirit
   defer, and to whom?

Output: a narrative report — what the spirit did at each step,
whether it passed, and the initiator's judgment. This is a
LabReport (dmt-eval pattern) applied to a spirit rather than a model.

* What This Proves

That verification of organisational artifacts is not infrastructure
bolted onto the bathhouse — it is a spirit in the bathhouse. The
critic archetype applied to the organisation itself, not just to
scientific models.

That the three-layer decomposition works for a spirit whose "project"
is the organisation: the power (birth-certificate, first-breath) is
portable; the skill binding is harness-specific; the task grounding
is "test this particular spirit in this particular project."

And that the Feynman imperative — "what I cannot create, I do not
understand" — applies to organisations as much as to physics. If you
cannot verify your own commissioning process, you do not understand
it. Dixa is the understanding made operational.

* Open Questions

- *Schema format*: the schemas in =nature/schemas/= could be YAML
  (simple, readable), JSON Schema (standard, tooling), or Python
  dataclasses (like dmt-eval's Protocol pattern). YAML feels right
  for an org that thinks in YAML — but JSON Schema has validation
  libraries. The human's preference for simplicity suggests YAML
  with a custom validator.

- *First-breath automation*: Phase 2 is human-administered for now.
  When WP-0005's dispatch machinery exists, dixa could spawn an
  agent session, run the scripted prompts, and collect the results
  automatically. This is the path from ritual to reflex — but the
  ritual should be understood before it's automated.

- *Retroactive initiation*: existing spirits (dmt-eval-guardian,
  aikosh-guardian, epistem-guardian, sutradhar-guardian) were
  commissioned before dixa existed. Should they undergo first-breath
  retroactively? The birth certificate yes (schema check is fast
  and informative). The first breath is optional — they've already
  proven viability through work.

- *Dixa's own initiation*: who initiates the initiator? This is the
  bootstrapping problem. Dixa passes its own birth certificate
  (self-validation). Its first breath is this WP — the act of
  building it /is/ the initiation. The guild accepts the proof of
  work, not the proof of test.

- *Where does dixa's system prompt live?*: dixa has no project
  CLAUDE.md because it operates across projects. Its "system prompt"
  is its identity.yaml + the contract field + the nature/ code.
  This is a new pattern — a spirit whose prompt /is/ its code.
