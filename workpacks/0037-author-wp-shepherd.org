* Context and Motivation

WP-0030 through WP-0036 tested substrate diversity on a /receptive/ task:
read the project, orient, produce notes. The orient-to-mayalucia power
asks substrates to /understand/. But understanding is half the loop.
The other half is /specification/ — translating understanding into an
actionable artifact.

This experiment tests the *author-wp* power across diverse cognitive
substrates. Each substrate receives the full orient-to-mayalucia
preamble (isolating author-wp as the variable), the same concrete
problem (spirit-guild cross-reference gaps in aburaya), and tool access
via gptel-agent.

The orchestrator is a *gaddi* — named for the nomadic shepherds of the
Dhauladhar and Pir Panjal who walk with their flocks across high
passes, reading terrain, knowing which route is open, accumulating
traversal knowledge across seasons. The gaddi does not own the pasture
at either end. It traverses. Each substrate traversal of the author-wp
task teaches the gaddi something about the power's portability.

The gaddi role is itself a testable substrate behavior. In this
experiment, mayadev#opus-4.6 is the gaddi. How it orchestrates,
monitors, injects prompts, and analyses — that is the Opus 4.6
contribution. A future experiment may test a different substrate as
gaddi on the same task.

** Why this matters

- *Orient* measures comprehension. *Author-wp* measures judgment:
  scoping, surveying, specifying, and knowing when to stop.
- A substrate that orients well may still produce vague or over-broad
  WPs. Or it may over-specify. The failure modes are different from
  orient failures.
- The spirit-guild gap is small enough to be tractable in a single
  session, yet real enough that the WP must be precise to be executable.

** The larger goal

This experiment is a step toward a *publishable, organisation-independent
onboarding tool*. The orient-to-mayalucia power already encodes
substrate-independent principles distilled from 11 lineages. The gaddi
pattern generalises: any organisation with structured documentation can
use the same orient → specify loop to onboard agents. The MayaLucIA
spirit-guild gaps are the test case. The tool is the product.

Token efficiency — tokens consumed per unit of organisational complexity
onboarded — is the quantitative measure that makes this publishable.
An organisation shouldn't need to know which substrate it's buying;
the tool should work for any of them.

** The gap (problem given to substrates)

Two cross-reference inconsistencies in =aburaya/=:

1. *percept-guardian-placeholder* is listed in =guilds/percept.yaml=
   spirits section, but no =spirits/percept-guardian-placeholder/identity.yaml=

Substrates must discover these gaps (or others they find), decide what

** 3. Metrics

*** Token efficiency

For each substrate, record:

| Metric                 | How to measure                                                  |
|------------------------+-----------------------------------------------------------------|
| Input tokens           | Total tokens consumed reading files + orient preamble + prompts |
| Output tokens          | Total tokens produced (tool calls + WP draft + reflection)      |
| Total tokens           | Input + output                                                  |
| Organisational surface | Count of files read, YAML keys parsed, cross-references checked |
| Token efficiency ratio | Total tokens / organisational surface                           |
| Cost (USD)             | Total tokens x per-model pricing from WP-0036 spec table        |

The token efficiency ratio measures how many tokens a substrate burns
per unit of organisational complexity it processes. Lower is better.
A substrate that reads 12 files and produces a precise WP in 8K tokens
is more efficient than one that reads 30 files and produces a vague WP
in 40K tokens.

* What This Proves

If the experiment works:

1. *author-wp is substrate-portable* — diverse lineages can exercise
   the power and produce recognisable WP artifacts
2. *Specification quality varies by lineage* — different substrates
   make different scoping and precision trade-offs
3. *Token efficiency is measurable* — we can quantify the cost of
   onboarding an agent to an organisation, per substrate
4. *The gaddi pattern works* — a spirit can orchestrate other
   substrates through a task more complex than orientation
5. *The gap itself gets documented* — at least one substrate should
   produce a WP good enough to actually execute
6. *The gaddi power is extractable* — the orchestration pattern can
   be written as a harness-neutral power document

If it fails:

- Some substrates may not be able to exercise author-wp (tool-call
  failures, inability to survey, convention blindness)
- The gptel harness may impose limits (Gemini thinking-mode conflicts,
  API timeouts, buffer management)
- The token efficiency metric may not capture meaningful variation
  (all substrates may converge on similar token counts)
- These failure modes are themselves data for the apprentis guild

* Extension: Phase 2 — Tool-Correct Pipeline with More Substrates

** Motivation

Phase 1 (Steps 0–15) completed with 3 substrates (Kimi K2.5, Gemini
3.1 Pro, Qwen 3.5+). All produced results but the gaddi could not run
fully autonomously because:

1. *Tool confirmation gates* — the Gemini session used filesystem tools
   (opencode-sourced) that required interactive approval from the human,
   outside gptel's =gptel-confirm-tool-calls= control. The gaddi stalled
   at each confirmation prompt.

2. *Tool registry mismatch* — gptel-agent's native tools (=gptel-agent-tools.el=)
   have =:confirm t= only on mutating tools (Bash, Edit, Write, Insert,
   Mkdir, Eval, Agent). Read-only tools (Read, Glob, Grep) have no
   =:confirm= slot and run freely under the =auto= setting. But if a
   session picks up tools from a /different/ registry (opencode), those
   tools carry their own confirmation mechanism that gptel cannot disable.

The fix is at configuration time: use a dedicated agent definition that
declares /only/ gptel-agent's native read-only tools.

** Surveyor agent

A new gptel-agent definition: =mayadevgenz/agents/surveyor.md=.

| Property       | Value                          |
|----------------+--------------------------------|
| Tools          | Read, Glob, Grep, TodoWrite    |
| Mutating tools | None                           |
| Confirm gates  | Zero (all tools are read-only) |
| Purpose        | Audit, survey, specify         |

The surveyor cannot edit files. Its output is text in the conversation
buffer — WP drafts, audit reports, reflections. This makes gaddi
orchestration fully autonomous: no tool confirmation, no human
intervention between prompts.

The system prompt encodes survey methodology (discover-then-interpret,
scope tightly, name concretely, write testable criteria) without
project-specific content. The orient-to-mayalucia preamble and task
brief are injected by the gaddi as the prompt sequence, same as Phase 1.

*** Post-session log extraction

The session buffer shows only the model's prose output. Tool calls,
token counts, and API-level metadata live in the =*gptel-log*= buffer.
After each substrate completes (both prompts answered), extract:

**** 1. Token counts (from the =usage= block in the final streaming chunk)

Each OpenRouter response ends with a =usage= JSON object:

#+begin_example
"usage": {
  "prompt_tokens": N,
  "completion_tokens": N,
  "total_tokens": N,
  "cost": N.NNNNN,
  "completion_tokens_details": { "reasoning_tokens": N }
}
#+end_example

Sum across all API round-trips for the session (one per model turn —
each tool-call cycle is a separate round-trip). Use =emacsclient= to
query the log buffer programmatically:

#+begin_src elisp :tangle no
;; Count tool invocations by name
(with-current-buffer "*gptel-log*"
  (save-excursion
    (goto-char (point-min))
    (let ((count 0))
      (while (search-forward "\"name\":\"Read\"" nil t)
        (setq count (1+ count)))
      count)))
#+end_src

**** 2. Tool usage profile

For each tool (Glob, Read, Grep, TodoWrite), count invocations.
This gives the /organisational surface/ — how many file operations the
substrate performed to produce its specification.

**** 3. Streaming errors

Count =*ERROR*: Unknown message:= entries in the session buffer.
These are gptel's streaming parser struggling with certain character
sequences in the model's output (e.g. YAML frontmatter, Org table
separators). They do not indicate API failures — the content still
arrives. Record the count as a harness friction metric.

**** 4. Save extracted data

Append a metrics block to the end of each session file:

#+begin_example
* Metrics (extracted from *gptel-log*)
#+property: extracted-by mayadev
#+property: extracted-date YYYY-MM-DD

| Metric                    | Value   |
|---------------------------+---------|
| API round-trips           | N       |
| Prompt tokens (total)     | N       |
| Completion tokens (total) | N       |
| Reasoning tokens (total)  | N       |
| Total tokens              | N       |
| Cost (USD)                | N.NNNNN |
| Glob calls                | N       |
| Read calls                | N       |
| Grep calls                | N       |
| TodoWrite calls           | N       |
| Streaming parse errors    | N       |
#+end_example

**** Important: do not inject into gptel during streaming

The =*gptel-log*= buffer is shared across all sessions. Reading it
via =emacsclient= while a substrate is streaming will not corrupt the
stream (read-only access is safe). But *never inject text into the
session buffer* while the model is streaming — this corrupts gptel's
response handler. Extract metrics only after =finish_reason: stop=
appears in the log for that session's final round-trip.

   simultaneously if tool gates are eliminated.

    sequence, the experiment is repeatable: any new lineage can be
