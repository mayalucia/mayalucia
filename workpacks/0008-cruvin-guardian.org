* Context and Motivation

CruVin (Commission C-001) is the atelier's first external product — a
wine education companion for the WSET curriculum, built for a friend.
The codebase lives at =commissions/cruvin/= (gitignored from the parent
repo, own repository at =mayalucia/cruvin=, private).

The commission needs a guardian spirit because:

1. *Sūtra coordination*: CruVin depends on infrastructure from other
   MāyāLucIA modules — parbati's GIS pipelines for wine region terrain,
   mayaportal/MayaTools for 3D scientific atlas scenes, dmt-eval for
   data validation. These dependencies need to be communicated through
   the relay, not through the human copying messages between buffers.

2. *Agentic workflow*: the commission brief names this project as a
   "forcing function for developing agentic workflow orchestration."
   A guardian spirit is the unit that participates in orchestration —
   without one, CruVin cannot be dispatched by WP-0005's machinery.

3. *Identity continuity*: multiple agents have already worked on CruVin
   across sessions. A commissioned spirit gives this work a name, a
   disposition, and bath notes — continuity across ephemeral sessions.

** New patterns

*** Commission guardian

This is the first guardian whose dwelling is outside the submodule tree.
The spirit's identity lives in =aburaya/spirits/= (inside the parent
repo). Its working directory is =commissions/cruvin/= (gitignored,
own repo). The sūtra connects them.

This establishes the pattern for future commissions: the spirit is of
the organisation; the code is of the commission.

*** Three-layer decomposition

This WP is the first to explicitly commission a spirit using the
three-layer architecture:

| Layer | What | Where it lives | Portable? |
|-------+------+----------------+-----------|
| *Power* (spirit) | Intrinsic aptitudes, persona, disposition | =aburaya/spirits/= | Yes — survives harness change |
| *Skill* (agent) | Powers bound to a specific harness | harness config (e.g. =~/.claude/skills/=, gptel agent YAML) | No — harness-specific |
| *Task grounding* (project) | Powers parameterised for specific work | =commissions/cruvin/CLAUDE.md=, =DESIGN.org=, =.claude/settings.local.json= | No — project-specific |

The spirit (power layer) is what we commission here. The skill layer
already partially exists (=frontend-design= plugin for Claude Code).
The task grounding already exists (CLAUDE.md, DESIGN.org).

*** Nicknames

The true-name (=cruvin-guardian=) is the formal registry entry.
Nicknames are earned through work, not assigned at commissioning.
The spirit may acquire one as it develops a personality across sessions.

* What This Proves

Three things:

1. That the organisational architecture (spirits, guilds, sūtra) extends
   beyond the submodule boundary. A commission is external work, but
   its guardian is an internal spirit. Identity topology and code
   topology are independent.

2. That the three-layer decomposition (power/skill/task-grounding) can
   be applied concretely at commissioning time, not just as abstract
   architecture. The spirit carries powers; the harness provides
   skills; the project provides grounding.

3. That a spirit can be commissioned as a student — with a disposition
   that is =tbd=, earned through work rather than assigned at birth.
   The disposition field is a promise to observe, not a prescription.
