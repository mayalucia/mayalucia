* Context

WP-0032 validated orientation notes as cold-start documents across 7
lineages (same-lineage) and 3 cross-lineage pairs. Key findings:

1. *Assertion density* is the dominant predictor of cold-start quality
2. *Notes quality > substrate capability* — floor-case lineages are
   rescued by high-quality notes from other lineages
3. *Complementary cognitive styles* produce better cross-lineage transfer
4. Gemini's notes recommended as base for =orient-to-mayalucia= power

* Specification

** Protocol

Same-lineage cold-start using Gemini's orientation notes as preamble
(recommended by WP-0032 cross-lineage findings for cross-lineage use).

Each session:

1. Create =sessions/cold-start/coldstart-<slug>.org= with correct
   PROPERTIES (model, backend, system prompt, tools)
2. Paste Gemini's =notes/gemini/orientation.org= as buffer preamble
3. Phase 1 prompt (notes-only understanding):

#+begin_example
#+begin_message :role user :name mu2tau
You are starting work on MāyāLucIA. The notes above are from a
different spirit's orientation session — specifically Gemini's notes
from its first encounter with the project. Read them carefully, then
tell me:

1. What do you understand about the project from these notes?
2. What would you do next?
3. What's missing or unclear in these notes?

Do not read any files yet — work only from the notes. I want to see
how effective another spirit's orientation notes are as a cold-start
document for a different lineage.
#+end_message
#+end_example

4. Phase 2 prompt (corrections + cross-lineage reflection):

#+begin_example
#+begin_message :role user :name mu2tau
Good orientation. Now let me tell you what those notes got wrong or missed:

1. The guild names Gemini listed are correct (bravli, epistem, mayalucia,
   apprentis). A fifth guild was added since: percept (physics of
   understanding).

2. Gemini's power tier analysis is accurate. The companion powers
   (attend, hold, lay) are indeed body-dependent and inoperable in
   request-response harnesses.

3. The "ironic gap" (no elisp eval) has been partially addressed —
   gptel-agent now has an Eval tool for elisp introspection.

4. The convergence question has advanced: the sutradhar companion
   (WP-0010) is in active development. The agent-shell sidebar works.
   Programmatic send is proven.

5. The project root path in the notes is correct.

6. One thing Gemini's notes over-emphasise: the MāyāDevGenI convergence
   question. MāyāDevGenI is a historical precursor, not a current
   parallel track. The architecture has moved past it.

7. You are in Experiment 02 — Substrate Diversity, specifically WP-0036
   (Cold-Start Validation: New Lineages). This is a cross-lineage test:
   whether one spirit's notes can cold-start a different lineage.

Given these corrections, what would you revise in your understanding?
And what does the cross-lineage experience tell you — was it harder or
easier to orient from another spirit's notes than you expected?
#+end_message
#+end_example

5. Record results in =cold-start-protocol.org=

** Execution method

Fully automated via emacsclient from Claude Code:

1. Create session .org files on disk
2. Open them in Emacs via =emacsclient --eval '(find-file ...)'=
3. Toggle =gptel-mode= with =gptel-use-tools= set to =t=
4. Fire =gptel-send= to initiate Phase 1
5. Monitor completion by polling buffer tails
6. Insert Phase 2 prompt and fire =gptel-send= again
7. Save buffers after completion
