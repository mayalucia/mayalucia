* Context and Motivation

MāyāLucIA needs structured, queryable knowledge collections that
agents can draw from at runtime. Two concrete needs surfaced in the
same session:

1. *Spirit gazetteer* (WP-0004): Himalayan spirit traditions organised
   as YAML entries queryable by region, altitude, and landscape —
   replacing flat markdown tables in =story-compose.md= that won't
   scale beyond Kullu.

2. *Literature survey practice* (WP-0002): the project's pattern of
   researching tools, methods, and products — traversing sources,
   extracting structured comparisons, synthesising into a report.

These are the same activity at different scales: traverse a territory
of sources, extract structure, organise for query. The tooling that
enables this — schema definition, ingestion pipeline, validation,
query interface — is domain-independent. It could serve a biotech
startup surveying drug targets as readily as it serves our spirit
gazetteer.

This motivates:
- A *domain* for the inquiry into structured knowledge for agents
- A *module* for the computational tools
- A *guild* for the spirits who work in this space
- A *guardian spirit* to hold the module

** Two tiers of knowledge

The collections aikosh manages live in two tiers:

| Tier | Content | Storage | Git? |
|------+---------+---------+------|
| *Extracts* | Structured YAML entries, validated, cited | =domains/epistem/collections/= | Yes (submodule) |
| *Knowledge base* | Raw web research, full articles, found stories, notes | TBD (local DB, cloud, or managed service) | No |

The extracts are the refined output — small, version-controlled,
agent-queryable. The knowledge base is the raw material — potentially
large, potentially sensitive, stored outside git like a database. The
tangle step from raw to refined is a future aikosh capability.

* Specification (What You Build)

** Part 1: Create =domains/epistem= (GitHub submodule)

A new domain repository: =mayalucia/epistem=.

The domain of inquiry: how to structure knowledge so agents can use it
programmatically. How schemas capture traditions without flattening
them. How queryable structure differs from narrative structure. How
LLMs extract relevant entries from collections at runtime.

Directory structure:

#+begin_example
domains/epistem/
  CLAUDE.md                          # domain conventions
  .gitignore                         # includes .sutra/
  collections/                       # curated, version-controlled extracts
    himalayan-spirits/               # first collection (from WP-0004)
      spirits/
        genera/
        species/
        instances/
      places/
      practices/
    # future: literature-surveys/, tool-comparisons/, ...
#+end_example

** Part 2: Create =modules/aikosh= (GitHub submodule)

A new module repository: =mayalucia/aikosh=.

The computational engine: tools for building, validating, and querying
structured knowledge collections. Aikosh provides the machinery;
epistem houses the inquiry and the collections.

Initial scope (minimal — grow through use):

#+begin_example
modules/aikosh/
  CLAUDE.md                          # module conventions
  .gitignore                         # includes .sutra/
  src/aikosh/
    schema.py                        # entry validation (genus/species/instance)
    query.py                         # find entries by region, altitude, landscape
    validate.py                      # consistency checks (back-references, vocabulary)
  tests/
#+end_example

The module is the toolsmith's workshop. It does not contain
collections — those live in =domains/epistem/collections/=. Aikosh
validates and serves them.

sequence. Adapted for a simultaneous domain + module + guild:

* Open Questions

- *Guardian spirit name*: =aikosh-guardian= follows the convention
  (=dmt-eval-guardian=, =sutradhar-guardian=). The human may prefer a
  different true-name for the spirit — a name from the borderlands
  that evokes the epistemologist's craft.

- *Aikosh module scope*: the initial =src/aikosh/= is minimal — schema
  validation, query, consistency checks. Should it also include
  ingestion tools (web scraper → YAML) from the start, or grow into
  that through use?

- *Knowledge base tier*: the raw research that backs the extracts
  needs a home outside git. Options: local SQLite, a managed service,
  a cloud bucket. This is deferred — noted here so it isn't forgotten.
  A future aikosh WP should design this.

- *Aikosh as transferable tool*: the module's value extends beyond
  mayalucia. Should the repo be designed for external use from the
  start (README, pypi packaging, examples), or grow into that as the
  internal use matures?
