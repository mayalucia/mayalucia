* Intent

Every spirit in the MāyāLucIA ecology currently assumes Claude Code as its
substrate. The =nature:= block in =identity.yaml= names a specific model; the
=character.system_prompt= points to =CLAUDE.md=; the agent-shell harness
hardcodes =agent-shell-anthropic-make-claude-client= for every spirit.

This workpackage separates what a spirit /is/ from what it /inhabits/. A spirit
is its character, role, disposition, and powers. A substrate is a model on a
provider, accessed through a specific CLI tool with its own system-prompt
conventions. The same spirit should be able to incarnate in Claude Code, Gemini
CLI, OpenAI Codex, or any future backend — with the harness resolving the
binding at session launch.

* Scope

Three layers of change:

1. *Identity schema* — refactor =.guardian/identity.yaml= across all 6 spirits
2. *System prompt architecture* — establish =system.md= (backend-neutral) with thin backend adapters
3. *Agent-shell harness* — modify =spirit-session.el= to resolve substrate at launch time

* Addressing Convention

A spirit's full address is =spirit@machine#substrate=:

#+begin_example
mayadev@vadda#claude-code      — full form
mayadev#gemini-cli             — machine elided (implied from session)
mayadev#gemini                 — short substrate name (implied mapping)
mayadev                        — default substrate elided
epistem-guardian#gemini         — non-default substrate shown
#+end_example

The =#= separator distinguishes substrate from host. The =@= separates
spirit from machine. Both are elidable — the intelligences reading these
addresses (human or LLM) infer the missing components from context.

In agent-shell buffer names, only the =spirit#substrate= part is used.
The default substrate is elided: =mayadev= when on claude (the default),
=mayadev#gemini= when on a non-default substrate.

* Design

** 1. Identity Schema Refactoring

*** Remove =nature:= from =identity.yaml=

The =nature:= block (model, provider, context_window, knowledge_cutoff)
describes the substrate, not the spirit. Remove it entirely from identity.yaml.

*** Remove substrate-specific tools from =dwelling:=

Tools like =claude-code=, =gemini-cli= belong to the substrate binding.
Universal tools (=hugo=, =python=, =uv=, =shadow-cljs=) stay in =dwelling.tools=.

*** Replace =character.system_prompt= with backend-neutral reference

Instead of =system_prompt: CLAUDE.md= (which is backend-specific), use:
#+begin_example
character:
  system_prompt: system.md
#+end_example

*** New schema (identity.yaml v2)

#+begin_src yaml
# .guardian/identity.yaml v2 — the spirit, substrate-free

true-name: <spirit-name>

dwelling:                            # station — the current host
  machine: <hostname>
  os: <os>
  shell: <shell>
  tools: [<universal-tools-only>]    # no claude-code, no gemini-cli
  repos:
    <name>: <path>

character:                           # personality — what persists across substrates
  archetype: <archetype>
  system_prompt: system.md           # backend-neutral prompt
  powers: [<list>]
  domains: [<list>]
  disposition: <disposition>

memory:                              # history — pointers, not content
  bath_notes: <path>
  provenance: >
    <origin-story>

project:                             # shared commons
  name: <project-name>
  repo: <repo-url>
  path: <relative-path>

role:                                # relational
  guild: <guild>
  function: <function>
  scope: [<list>]
  contract: >
    <optional-contract>
#+end_src

  substrates/
    claude.yaml         # Claude Code binding
    gemini.yaml         # Gemini CLI binding
    codex.yaml          # OpenAI Codex binding (future)
#+end_example

*** Default substrate

When no substrate is specified, the harness reads =.guardian/substrates/default=
(a plain text file containing e.g. =claude=). If absent, falls back to =claude=.

** 3. System Prompt Architecture

*** The problem

Each backend has its own system-prompt conventions:
- Claude Code reads =CLAUDE.md= — supports skill references, tool-use XML
- Gemini CLI reads =GEMINI.md= — uses =@file= inclusion, google-specific tool names
- Codex reads =AGENTS.md= (or similar)

Spirit identity (collaborative stance, project structure, conventions, role
contract) is the same across backends. Backend-specific instructions (tool
names, output conventions, context-window strategy) differ.

*** The solution: =system.md= + backend adapters

Each spirit directory gets:

#+begin_example
  system.md       # Backend-neutral spirit prompt (identity, stance, conventions)
  CLAUDE.md       # Thin adapter: includes system.md content + Claude-specific instructions
  GEMINI.md       # Thin adapter: includes system.md content + Gemini-specific instructions
#+end_example

**** =system.md= contains

- Identity section (reads from identity.yaml)
- Collaborative stance (The Sculptor's Paradox, Feynman Imperative, etc.)
- Project structure and conventions
- Role contract
- Powers description
- Sūtra protocol

**** Backend adapters contain

- An instruction to read/incorporate =system.md=
- Backend-specific tool execution rules
- Context-window strategy (e.g. Gemini's 2M token synthesis mandate)
- Backend-specific file naming conventions
- Authentication/environment notes

**** Migration for existing CLAUDE.md files

The current CLAUDE.md files contain both spirit-universal and Claude-specific
content interleaved. For each spirit:

1. Extract the universal content into =system.md=
2. Reduce =CLAUDE.md= to a thin adapter that references =system.md= and adds
   Claude-specific rules
3. Create =GEMINI.md= adapter where relevant (initially: root, epistem)

*** Adapter reference patterns

For Claude Code (which auto-reads CLAUDE.md):
#+begin_example
# CLAUDE.md — Claude Code adapter for <spirit-name>

Read and follow all instructions in system.md in this directory. That file
defines your identity, collaborative stance, and project conventions.

#+end_example

For Gemini CLI (which auto-reads GEMINI.md):

#+end_example

** 4. Agent-Shell Harness Changes

*** =agent-shell--guardian-substrate=

New function: read substrate from =.guardian/substrates/=.

#+begin_src elisp
(defun agent-shell--guardian-substrate (&optional backend)
  "Read substrate config for BACKEND from .guardian/substrates/.
BACKEND is a string like \"claude\", \"gemini\", \"codex\".
If nil, read .guardian/substrates/default, falling back to \"claude\"."
  (when-let* ((root (agent-shell--guardian-root)))
    (let* ((backend (or backend
                       (let ((default-file (expand-file-name
                                            ".guardian/substrates/default" root)))
                         (if (file-exists-p default-file)
                             (string-trim
                              (with-temp-buffer
                                (insert-file-contents default-file)
                                (buffer-string)))
                           "claude"))))
           (substrate-file (expand-file-name
                            (format ".guardian/substrates/%s.yaml" backend)
                            root)))
      (when (file-exists-p substrate-file)
        (cons backend
              (with-temp-buffer
                (insert-file-contents substrate-file)
                (buffer-string)))))))
#+end_src

*** =agent-shell--client-maker-for-backend=

Dispatch to the correct client-maker based on substrate.

#+begin_src elisp
(defun agent-shell--client-maker-for-backend (backend)
  "Return a client-maker lambda for BACKEND string."
  (pcase backend
    ("gemini" (lambda (buffer)
               (agent-shell-google-make-gemini-client :buffer buffer)))
    ("codex"  (lambda (buffer)
               (agent-shell-openai-make-codex-client :buffer buffer)))
    (_        (lambda (buffer)
               (agent-shell-anthropic-make-claude-client :buffer buffer)))))
#+end_src

*** Modify =agent-shell--make-spirit-config=

Read substrate and dispatch accordingly. The buffer name uses the
=spirit#substrate= notation (e.g. =mayadev#gemini=) so that multiple
incarnations of the same spirit can coexist. The default substrate
is elided: =mayadev= means =mayadev#claude= when claude is the default.

#+begin_src elisp
(defun agent-shell--make-spirit-config (name &optional backend)
  "Build agent-config for spirit NAME on BACKEND.
BACKEND is a string (\"claude\", \"gemini\", \"codex\") or nil for project default.
Buffer name uses spirit#substrate notation; default substrate is elided."
  (let* ((substrate (agent-shell--guardian-substrate backend))
         (resolved-backend (or (car substrate) "claude"))
         (default-p (agent-shell--default-substrate-p resolved-backend))
         (buf-name (if default-p
                       name
                     (format "%s#%s" name resolved-backend)))
         (color (agent-shell--spirit-color name))
         (prompt (propertize (concat buf-name "> ")
                             'face `(:foreground ,color :weight bold))))
    (agent-shell-make-agent-config
     :identifier (intern buf-name)
     :buffer-name buf-name
     :mode-line-name buf-name
     :shell-prompt prompt
     :shell-prompt-regexp (concat (regexp-quote buf-name) "> ")
     :welcome-function (agent-shell--spirit-welcome name)
     :client-maker (agent-shell--client-maker-for-backend resolved-backend))))

(defun agent-shell--default-substrate-p (backend)
  "Return non-nil if BACKEND is the project's default substrate."
  (let* ((root (agent-shell--guardian-root))
         (default-file (when root
                         (expand-file-name ".guardian/substrates/default" root)))
         (default (if (and default-file (file-exists-p default-file))
                      (string-trim
                       (with-temp-buffer
                         (insert-file-contents default-file)
                         (buffer-string)))
                    "claude")))
    (string-equal backend default)))
#+end_src

*** Modify =agent-shell-start-spirit= for backend selection

=C-u SPC o g= prompts for backend in addition to forcing a new session.

#+begin_src elisp
(defun agent-shell-start-spirit (&optional name)
  "Start agent-shell with spirit identity.
Resolve NAME by: explicit argument → .guardian/identity → interactive prompt.
With prefix arg, prompt for backend and force a new session."
  (interactive)
  (let* ((resolved (or name
                      (agent-shell--guardian-name)
                      (let ((input (read-string "Spirit (empty for default): ")))
                        (unless (string-empty-p input) input))))
         (backend (when current-prefix-arg
                    (completing-read "Backend: "
                                    (agent-shell--available-substrates)
                                    nil t nil nil "claude")))
         (config (if resolved
                     (agent-shell--make-spirit-config resolved backend)
                   agent-shell-default-config)))
    (let ((existing (seq-find
                     (lambda (buf)
                       (string-equal
                        (map-nested-elt
                         (buffer-local-value 'agent-shell--state buf)
                         '(:agent-config :buffer-name))
                        (map-elt config :buffer-name)))
                     (agent-shell-project-buffers))))
      (if (and existing (not current-prefix-arg))
          (agent-shell--display-buffer existing)
        (agent-shell--start :config config)))))
#+end_src

*** Helper: list available substrates for a project

#+begin_src elisp
(defun agent-shell--available-substrates ()
  "List available substrate names from .guardian/substrates/*.yaml."
  (when-let* ((root (agent-shell--guardian-root))
              (dir (expand-file-name ".guardian/substrates/" root))
              ((file-directory-p dir)))
    (mapcar #'file-name-sans-extension
            (directory-files dir nil "\\.yaml\\'"))))
#+end_src

* Risks and Open Questions

** Prompt inclusion mechanism differs per backend

Claude Code auto-reads =CLAUDE.md= from the working directory. Gemini CLI
auto-reads =GEMINI.md=. Codex reads =AGENTS.md=. The =system.md= file is
/not/ auto-read by any backend — it must be referenced from the adapter file.

This means =system.md= is a /source/ file, not a /runtime/ file. Each backend
adapter must explicitly include or reference it. Verify that each backend's
inclusion mechanism (Gemini's =@./system.md=, Claude's manual read instruction)
actually works in practice.

** Codex and future backends

OpenAI Codex CLI, MiniMax, DeepSeek, Kimi — we don't have substrate files or
adapters for these yet. The schema supports them (add a =.yaml= to substrates,
add a case to the =pcase= dispatch). But we should not create speculative
substrate files — only create them when we actually configure the backend.

** The =.guardian/spirit-session.el= dual-source problem

Currently =spirit-session.el= lives both in =.guardian/= (canonical) and
inline in =~/.doom.d/config.org= (tangled). Phase 3 must update both.
Consider whether config.org should simply =(load ...)= the canonical file
instead of duplicating it.

** Sidebar substrate

The sūtradhar sidebar currently hardcodes Claude. Should it follow the
sutradhar module's default substrate, or should the sidebar always be
Claude (the Max subscription)? The sidebar is the human's persistent
companion — the substrate choice may be a UX preference, not a project
concern. Leave as Claude for now, note as future decision.
